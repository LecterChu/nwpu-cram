## 第 3 章：UML 建模

第 3 章是综合设计题的主战场。你要会把一个业务场景转成用例、类、消息和状态。

### 3.1 用例建模

用例是参与者和系统之间的一组交互序列，用来描述功能需求。系统在用例建模中被看成黑盒，只描述外部交互，不描述内部实现。

| 名词 | 解释 | 考试提醒 |
|---|---|---|
| 参与者 Actor | 与系统交互的外部实体，可以是人、外部系统、设备、计时器 | 主参与者获得价值，次参与者提供服务 |
| 系统边界 | 区分系统内部和外部环境 | 框内是用例，框外是参与者 |
| 用例 | 对参与者有价值的一组交互 | 用动词加名词命名，如“预约座位” |
| 用例说明 | 用文字描述一个用例完整过程 | 写前置、后置、主流程、备选流程 |

include、extend、泛化最容易考：

| 关系 | 含义 | 方向 | 例子 |
|---|---|---|---|
| include | 必须执行的公共子流程 | 基用例 -> 被包含用例 | “预约座位” include “验证登录” |
| extend | 满足条件才发生的可选或异常流程 | 扩展用例 -> 基用例 | “支付失败处理” extend “在线支付” |
| 泛化 | 子用例继承父用例并可特化 | 子 -> 父 | “微信支付”泛化“支付” |

用例说明模板：

```text
用例名称：
参与者：
前置条件：
后置条件：
主成功场景：
1. ...
2. ...
备选/异常场景：
3a. ...
```

### 3.2 静态建模：EBCA 类

静态建模回答：系统里有哪些类，它们是什么职责，它们有什么关系。

| 类型 | 职责 | 例子 |
|---|---|---|
| Entity 实体类 | 保存长期存在或可持久化的业务数据 | User、Seat、Reservation、Order |
| Boundary 边界类 | 连接用户、外部系统、设备 | BookingUI、PaymentGateway、DoorAdapter |
| Control 控制类 | 协调用例流程，安排对象调用顺序 | BookingController、OrderController |
| Application Logic 应用逻辑类 | 封装可复用业务规则或算法 | ConflictChecker、FeeCalculator、TimeoutManager |

控制类和应用逻辑类的区别：

```text
控制类管流程：先做什么、后做什么、异常时转到哪里。
应用逻辑类管规则：怎么判断冲突、怎么计算费用、怎么计算路线。
```

类之间常见关系：

| 关系 | UML 表示 | 含义 | 例子 |
|---|---|---|---|
| 关联 | 实线 | 普通联系 | User 关联 Reservation |
| 聚合 | 空心菱形 | 弱整体-部分，部分可独立存在 | StudyRoom 聚合 Seat |
| 组合 | 实心菱形 | 强整体-部分，生命周期绑定 | Order 组合 OrderItem |
| 泛化 | 空心三角 | 继承，is-a | CheckingAccount 继承 Account |

上下文类图用于划定系统边界。系统外部包括人类参与者、外部系统、设备、计时器。软件系统上下文类图中，I/O 设备通常画在软件系统外部。

### 3.3 动态建模：通信图、序列图、状态图

通信图和序列图都描述对象交互。

| 图 | 重点 | 优点 | 缺点 |
|---|---|---|---|
| 通信图 | 对象之间的连接关系，消息用编号表示 | 适合看对象拓扑，课件实验常用 | 时间顺序靠编号，不如序列图直观 |
| 序列图 | 消息按时间从上到下排列 | 适合看流程、并发和返回 | 对象关系不如通信图突出 |

动态建模固定套路：

```text
参与者 -> 边界类 -> 控制类 -> 应用逻辑类/服务类 -> 实体类或外部系统
```

状态图用于对象有明显生命周期时，例如订单、预约、报警、AGV 车辆、智能体任务。

| 状态图元素 | 含义 |
|---|---|
| 状态 | 对象在某一阶段的稳定情况，如“待支付” |
| 事件 | 触发状态变化的事情，如“支付成功” |
| 条件 | 守卫条件，事件发生且条件为真才迁移 |
| 动作 | 状态迁移时执行的操作 |

E-C-A 是 Event、Condition、Action。写法是：

```text
事件[条件]/动作
```

SDCO 是状态相关控制对象。当控制对象的行为依赖当前状态时，它就是 SDCO。通信图中，进入 SDCO 的消息常是事件；SDCO 发出的消息常是动作；状态图说明条件和状态迁移。

## 第 4 章：软件架构模式

架构模式是反复出现的架构设计问题的可复用解决方案。常考定义、适用场景、优缺点。

### 4.1 软件架构和多视图

软件架构是系统软件元素、外部可见属性以及元素之间关系组成的高层结构。

| 视图 | 关注点 | 常用图 |
|---|---|---|
| 结构视图 | 子系统、构件、类之间的静态关系 | 类图、构件图 |
| 动态视图 | 运行时消息、控制流、状态变化 | 通信图、序列图、状态图 |
| 部署视图 | 软件部署到哪些硬件或网络节点 | 部署图 |

多视图的意义：单一图无法同时表达结构、行为和物理部署，所以需要从不同角度描述同一系统。

### 4.2 常见架构模式速查

| 模式 | 核心思想 | 优点 | 缺点或适用限制 |
|---|---|---|---|
| 分层模式 | 按职责分层，上层调用下层 | 职责清晰、易维护、易替换 | 层次过多会增加开销 |
| 调用/返回 | 调用方请求并等待结果 | 简单、结果明确 | 耦合紧、调用方阻塞 |
| 异步消息 | 发送者发消息后继续执行 | 解耦、削峰、适合分布式 | 一致性和错误处理更复杂 |
| 管道-过滤器 | 数据依次经过多个过滤器转换 | 处理步骤独立、可复用、可并行 | 不适合强交互业务 |
| MVC | Model、View、Controller 分离 | UI 和业务分离，多视图共享模型 | 小系统可能过度设计 |
| C/S | 客户端请求，服务器提供服务 | 职责分离、服务集中管理 | 服务器可能成为瓶颈 |
| SOA | 能力封装成服务，通过标准接口协作 | 松耦合、可复用、适合异构集成 | 服务治理和分布式事务复杂 |
| 构件架构 | 系统由构件和接口连接组成 | 接口与实现分离，可替换、可复用 | 构件划分和接口设计要求高 |

### 4.3 通信模式怎么选

| 场景 | 通信模式 | 理由 |
|---|---|---|
| 用户提交请求，需要立即反馈 | 同步请求/响应 | 用户等待结果 |
| 支付回调、短信通知 | 异步消息 | 外部处理时间不确定 |
| 一个状态变化通知多个界面 | 发布/订阅 | 一对多，降低耦合 |
| 高频传感器上报 | 消息队列 | 削峰、缓冲、提高吞吐 |
| AGV 紧急停车 | 高优先级广播 | 必须快速通知所有车辆 |

## 第 5 章：软件子系统架构设计

第 5 章连接 UML 分析和真正架构设计。核心问题是：大系统怎么拆成子系统，子系统怎么通信，怎么部署。

### 5.1 子系统是什么

子系统是系统中的较大模块，通常封装一组相关对象、服务或构件，对外通过接口提供能力。

子系统设计的目标是降低复杂度，使系统具有高内聚、低耦合、接口清晰、便于部署和维护。

### 5.2 关注点分离

| 划分依据 | 解释 | 例子 |
|---|---|---|
| 功能职责 | 相关功能放一起 | 用户、订单、支付 |
| 地理位置 | 不同地点对象分开 | 现场传感器和监控中心 |
| 服务器职责 | 不同服务器对应不同子系统 | Web、应用、数据库 |
| 外部接口 | 外部系统适配独立出来 | 支付网关、门禁接口 |
| 控制范围 | 控制不同状态机或任务 | AGV 控制、调度控制 |
| 变化率 | 经常变化和稳定部分分开 | UI 和核心业务规则 |

### 5.3 子系统接口

| 接口 | 含义 |
|---|---|
| 供给接口 Provided | 子系统对外提供什么操作 |
| 请求接口 Required | 子系统需要外部提供什么操作 |

接口说明应包括：操作名称、输入参数、输出参数、返回值、前置条件、后置条件、异常、通信协议。

### 5.4 部署图

部署图描述软件元素和硬件节点的映射关系。常见节点包括客户端、应用服务器、数据库服务器、消息中间件、外部系统或设备节点。

考试时至少画 2 到 3 个节点，标出关键子系统部署在哪个节点上，再说明这样部署如何影响性能、可用性和安全。

## 第 6-10 章：非重点概念补充

这部分不是老师点名重点，但可能作为名词解释、简答背景或综合题中的架构选择理由出现。只背概念、意义、优缺点，不要展开成大题。

### 第 6 章：面向对象的软件架构设计

| 概念 | 一句话定义 | 意义或优缺点 |
|---|---|---|
| MDA 模型驱动架构 | 从 CIM 到 PIM 再到 PSM，通过模型转换推动开发 | 提高可追踪、可移植、可复用；缺点是依赖模型质量和工具 |
| CIM / PIM / PSM | CIM 描述业务需求，PIM 抽象平台，PSM 面向具体技术平台 | 体现从需求到设计再到实现的逐步细化 |
| 信息隐藏 | 隐藏可能变化的数据结构和实现细节，只暴露稳定接口 | 提高可维护性和可修改性；接口设计差会变臃肿 |
| 主动对象 | 拥有自己的控制线程，可并发执行 | 适合并发/实时系统；要处理同步、互斥、死锁 |
| 被动对象 | 没有独立线程，被调用时才执行 | 简单、安全；并发能力弱 |
| 包装器类 | 封装遗产系统、数据库或外部接口 | 可复用旧系统；可能成为性能瓶颈 |
| 设计模式 | 经过验证的常见设计问题解决方案 | 复用经验；套用不当会过度设计 |

第 6 章答题句：

```text
面向对象架构设计强调职责分配、信息隐藏、接口稳定和对象协作，通过模型逐步细化把分析类转化为设计类。
```

### 第 7 章：客户端/服务器软件架构设计

| 概念 | 一句话定义 | 意义或优缺点 |
|---|---|---|
| C/S 架构 | 客户端请求服务，服务器提供共享资源或业务服务 | 职责分离、集中管理；服务器可能成为瓶颈或单点 |
| 胖客户端 | 客户端承担较多业务逻辑 | 服务器压力小；客户端维护困难 |
| 瘦客户端 | 客户端主要负责界面，业务逻辑在服务器 | 易维护；服务器压力大 |
| 顺序性服务 | 服务器一次只处理一个请求 | 实现简单、一致性容易；吞吐低 |
| 并发服务 | 服务器并发处理多个请求 | 吞吐高、响应快；同步和一致性更复杂 |
| 有状态服务 | 服务器保存客户端会话状态 | 交互方便；扩展和故障恢复困难 |
| 无状态服务 | 每个请求包含足够信息，服务器不保存会话 | 易扩展、易负载均衡；请求可能更重 |

第 7 章答题句：

```text
C/S 架构适合服务集中管理的系统，设计时要说明客户端、服务器、数据库的职责，以及服务器并发、状态保存和故障风险。
```

### 第 8 章：SOA 设计

| 概念 | 一句话定义 | 意义或优缺点 |
|---|---|---|
| SOA | 以自治服务为基本单元，通过标准接口协作 | 松耦合、可复用、适合异构集成；服务治理复杂 |
| 服务契约 | 服务对外公开的接口、参数、返回值、异常和协议 | 保证调用方和提供方按约定协作 |
| 服务粒度 | 服务能力划分的粗细程度 | 太粗不灵活，太细调用复杂 |
| 服务注册/发现 | 服务提供者注册，消费者通过注册中心查找服务 | 支持位置透明和动态绑定 |
| 代理者模式 | 客户端通过代理者访问服务 | 隐藏服务位置和平台差异；代理者可能成为瓶颈 |
| 编曲 Orchestration | 由一个中心流程控制服务调用顺序 | 易监控；中心协调者负担重 |
| 编舞 Choreography | 多个服务通过事件或协议自行协作 | 低耦合；全局流程难监控 |
| 两阶段提交 | 分布式事务原子提交协议 | 保证一致性；阻塞、性能低、协调者可能单点 |

第 8 章答题句：

```text
SOA 的核心是把业务能力封装成自治服务，通过标准接口、服务契约和服务组合实现松耦合复用。
```

### 第 9 章：基于构件的软件架构设计

| 概念 | 一句话定义 | 意义或优缺点 |
|---|---|---|
| 构件 Component | 有明确接口、可独立部署或组合的软件单元 | 可替换、可复用；接口设计要求高 |
| 供给接口 Provided Interface | 构件对外提供的服务 | 表示“我能提供什么” |
| 请求接口 Required Interface | 构件运行时需要外部提供的服务 | 表示“我需要什么” |
| 端口 Port | 构件与外部交互的连接点 | 把交互位置和接口绑定起来 |
| 连接器 Connector | 连接构件之间请求接口和供给接口的机制 | 明确构件协作关系 |
| 构件替换 | 用新构件替代旧构件而不影响其他部分 | 前提是接口兼容 |
| 集成通信图 | 多个用例通信图叠加后的对象协作图 | 用于发现子系统、构件和接口 |

第 9 章答题句：

```text
构件架构强调接口与实现分离，系统由构件、端口、供给接口、请求接口和连接器组合而成。
```

### 第 10 章：并发和实时软件架构设计

| 概念 | 一句话定义 | 意义或优缺点 |
|---|---|---|
| 实时系统 | 必须在时间约束内响应事件的系统 | 适合控制、调度、传感器系统；验证难 |
| 硬实时 | 超时会导致严重后果 | 必须严格保证时限 |
| 软实时 | 偶尔超时可接受但会降低质量 | 更关注平均响应和用户体验 |
| 事件流 | 系统连续接收的外部事件序列 | 实时架构要处理多个事件流 |
| 任务 Task | 并发执行的工作单元 | 需要定义优先级、周期、触发条件 |
| 互斥 | 多任务访问共享资源时排他控制 | 防止数据竞争；可能带来阻塞 |
| 死锁 | 多个任务互相等待资源导致无法继续 | 设计时要避免循环等待 |
| TIS | 任务接口规约，描述任务输入、输出、时序、优先级 | 说明任务之间如何交互 |
| TBS | 任务行为规约，描述任务如何响应事件 | 说明任务内部行为 |
| 集中式控制 | 一个控制器统一决策 | 简单；有单点和瓶颈 |
| 分布式控制 | 多个控制对象各自决策 | 自治性强；全局协调难 |
| 层次化控制 | 上层协调者管理多个局部控制对象 | 兼顾集中协调和局部自治 |

第 10 章答题句：

```text
并发实时架构的重点是事件、任务、状态和时间约束。设计时要说明主动对象、任务优先级、同步互斥、状态迁移和异常处理。
```

## 第 13 章：大数据软件架构

大数据是老师点名的新重点，重点是基本概念和典型架构，不需要背成大数据课程。

### 13.1 大数据 5V

| 特征 | 英文 | 含义 |
|---|---|---|
| 大量 | Volume | 数据规模大 |
| 高速 | Velocity | 数据产生和处理速度快 |
| 多样 | Variety | 结构化、半结构化、非结构化都有 |
| 价值 | Value | 数据需要被分析后产生价值 |
| 真实性 | Veracity | 数据质量、可信度、噪声问题 |

### 13.2 大数据分层架构

```text
数据源 -> 采集层 -> 消息层 -> 存储层 -> 计算层 -> 服务层 -> 展示/应用层
```

| 层 | 作用 | 常见组件 |
|---|---|---|
| 采集层 | 收集日志、业务数据、传感器数据 | Flume、CDC、日志采集 |
| 消息层 | 缓冲、削峰、解耦 | Kafka |
| 存储层 | 保存历史或明细数据 | HDFS、HBase、数据湖、数据仓库 |
| 计算层 | 批处理、流处理、查询分析 | Spark、Flink |
| 服务层 | 对外提供查询和指标 | API、Redis、TSDB |
| 展示层 | 报表、监控、告警 | Grafana、监控平台 |

### 13.3 Lambda 和 Kappa

| 架构 | 核心思想 | 优点 | 缺点 |
|---|---|---|---|
| Lambda | 批处理层 + 速度层 + 服务层 | 兼顾历史准确性和实时性 | 两套代码路径，维护复杂 |
| Kappa | 只保留流处理，历史重算也重放事件流 | 架构简单，只维护一套逻辑 | 依赖消息流保存和重放能力 |

Lambda 三层：

```text
批处理层：全量历史数据，结果准确但延迟高。
速度层：实时增量数据，结果低延迟但可能近似。
服务层：合并批处理视图和实时视图，对外查询。
```

### 13.4 数据湖、数据仓库、批流一体

| 名词 | 定义 | 背法 |
|---|---|---|
| 数据湖 | 存放原始、多格式数据 | 先存下来，再按需分析 |
| 数据仓库 | 面向主题、清洗后的结构化数据 | 面向报表和分析 |
| 批流一体 | 批处理和流处理统一 | 降低两套系统维护成本 |

### 13.5 PPT 案例里的组件

| 组件 | 作用 |
|---|---|
| Kafka | 高吞吐消息队列，负责流式接入、削峰、分区消费 |
| Redis | 内存缓存，适合实时轨迹匹配、热点数据查询 |
| InfluxDB/TSDB | 时序数据库，适合按时间保存监控指标 |
| CDC | 捕获数据库变更，用于新旧库同步或实时数据集成 |
| ETL | 抽取、清洗、转换、加载数据 |
| HDFS | 分布式文件系统，存储海量数据 |
| HBase | 面向列族的大规模明细查询 |
| Spark/Flink | 大规模计算，Spark 偏批和微批，Flink 偏原生流处理 |

大数据答题模板：

```text
数据从业务系统、日志或传感器采集，经 Kafka 缓冲和解耦。
实时数据由流处理组件计算，结果写入 Redis 或时序数据库用于实时查询和告警。
历史数据进入 HDFS/HBase 或数据仓库，用于离线分析和追溯。
```

## 第 14 章：AI 大模型与智能体架构

这一章是老师点名的新重点。核心是知道大模型、RAG、Agent、工具调用怎样组合成软件架构。

### 14.1 LLM 基本概念

LLM 是大语言模型，能理解和生成自然语言，也可用于代码、规划、推理、信息抽取等任务。

在软件架构中的定位：

```text
传统软件负责确定性业务逻辑。
LLM 适合处理自然语言理解、知识问答、推理生成、任务规划等不确定性任务。
现代系统常采用传统软件 + LLM 的混合架构。
```

### 14.2 Prompt 工程

| 技巧 | 含义 |
|---|---|
| 明确角色 | 告诉模型扮演什么角色 |
| 给出任务 | 说明要输出什么 |
| 给出约束 | 限制格式、范围、风格 |
| 给例子 | 用 few-shot 示例降低歧义 |
| 分步推理 | 让模型先分析再输出 |

### 14.3 RAG

RAG 是检索增强生成。它在大模型回答前，先从外部知识库检索相关内容，再把检索结果放进提示词。

```text
原始文档 -> 分段 -> 向量化 -> 向量库
用户问题 -> 向量化 -> 检索 TopK 文档片段 -> 组装 Prompt -> LLM 生成答案
```

作用：减少幻觉，接入私有知识，缓解模型知识过时问题。

### 14.4 AI Agent

智能体是能感知任务、规划步骤、调用工具并执行行动的软件实体。

| 模块 | 作用 |
|---|---|
| 感知 Perception | 接收用户输入、环境状态、工具反馈 |
| 规划 Planning | 分解目标，决定下一步 |
| 行动 Action | 调用工具、API、数据库或设备 |
| 记忆 Memory | 保存上下文、经验、知识 |

ReAct 循环：

```text
Thought 思考 -> Action 行动 -> Observation 观察 -> 再思考 -> 最终答案或行动结果
```

### 14.5 Function Calling 和多智能体

Function Calling 是让大模型按预定义 schema 调用外部函数或工具。

架构意义：

```text
大模型负责理解和规划，传统工具负责确定性执行。
这样既能利用模型的灵活性，又能保持业务系统的可靠性。
```

多智能体协作：

| 模式 | 适用场景 |
|---|---|
| 层级式 | 管理者 Agent 分解任务，工作者 Agent 执行 |
| 对等式 | 多个 Agent 平等讨论、评审、投票 |
| 专家式 | 不同 Agent 扮演不同专业角色 |

### 14.6 AI 架构答题模板

```text
系统可以采用传统业务系统 + AI 智能体的混合架构。
用户请求先进入智能体，智能体进行任务理解和规划；
需要知识时通过 RAG 检索知识库，需要行动时通过 Function Calling 调用业务 API 或工具。
大模型负责推理和生成，业务服务负责确定性执行，权限控制负责限制敏感操作。
```
