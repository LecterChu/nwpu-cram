## 一、简答题（5题，每题4分，共20分）

### 1、简述数据库、数据库管理系统和数据库系统的概念

**参考答案**：

| 术语                | 定义                                  |
| ----------------- | ----------------------------------- |
| **数据库（DB）**       | 长期存储在计算机内的、有组织的、可共享的大量数据的集合。        |
| **数据库管理系统（DBMS）** | 位于用户与操作系统之间的数据管理软件，负责对数据库进行统一管理和控制。 |
| **数据库系统（DBS）**    | 由数据库、数据库管理系统、应用程序和数据库管理员组成的整体。      |

> 三者关系：数据库系统 = 数据库 + 数据库管理系统 + 应用程序 + 管理员

---

### 2、简述关系模型的特点

**参考答案**：

关系模型具有以下特点：

1. **数据结构单一**：使用二维表（关系）表示数据和数据之间的联系。
2. **基于集合论**：以关系代数、关系演算为理论基础。
3. **数据独立性高**：逻辑独立性和物理独立性较好。
4. **支持完整性约束**：包括实体完整性、参照完整性和用户定义完整性。
5. **操作简单、非过程化**：用户只需说明“做什么”，不需说明“怎么做”。
6. **关系必须规范化**：基本要求是满足第一范式（1NF）。

---

### 3、简述事务的 ACID 特性

**参考答案**：

ACID 是保证数据库事务正确执行的四个特性：

| 特性                   | 含义                   | 作用               |
| -------------------- | -------------------- | ---------------- |
| **原子性（Atomicity）**   | 事务中的操作要么全部执行，要么全部不执行 | 通过恢复协议实现         |
| **一致性（Consistency）** | 事务执行前后数据库保持完整性约束     | 通过完整性约束、触发器实现    |
| **隔离性（Isolation）**   | 并发事务之间互不干扰           | 通过并发控制协议（如两段锁）实现 |
| **持久性（Durability）**  | 事务一旦提交，修改永久保存        | 通过恢复协议（日志、WAL）实现 |

---

### 4、简述视图概念以及通过视图访问数据库中数据的优缺点

**参考答案**：

**视图概念**：

- 视图是一个**虚拟表**，其内容由查询定义，不实际存储数据。
- 对视图的操作会转换为对基表的操作。

**优点**：

1. **数据独立性**：应用程序不依赖具体表结构。
2. **安全性**：可隐藏敏感数据，只暴露必要字段。
3. **简化查询**：将复杂查询封装为视图，便于使用。
4. **数据定制化**：为不同用户提供不同数据视图。

**缺点**：

1. **更新受限**：多表连接视图、含分组/聚合的视图不可更新。
2. **性能问题**：每次查询视图都需要执行定义查询，可能影响效率。
3. **结构限制**：视图不能索引（物化视图除外）。

### 5、简述应用程序通过 JDBC 方式连接数据库的过程

**参考答案**：

JDBC 连接数据库通常包含以下 **6 个步骤**：

| 步骤  | 操作            | 说明                                                 |
| --- | ------------- | -------------------------------------------------- |
| ①   | **加载驱动程序**    | `Class.forName("com.mysql.cj.jdbc.Driver")`        |
| ②   | **建立连接**      | `DriverManager.getConnection(url, user, password)` |

| ④   | **执行 SQL 语句** | `executeQuery()`（查询）或 `executeUpdate()`（增删改）       |

| ⑥   | **关闭资源**      | 关闭 `ResultSet`、`Statement`、`Connection`            |

**简化版（4分答案）**：

> ① 加载驱动 → ② 获取连接 → ③ 创建语句对象 → ④ 执行SQL → ⑤ 处理结果 → ⑥ 关闭资源

## 第6题

> 两段锁协议能够保证并发事务调度冲突可串行化的前提条件是 **每个事务都遵循两段锁协议**。

**解析**：

- 两段锁协议（2PL）要求每个事务的**加锁阶段**和**解锁阶段**不相交。
- 定理：若**所有并发事务都遵循两段锁协议**，则它们的**任何调度**都是冲突可串行化的。
- 反之，只要有一个事务不遵守，就不能保证。

**完整表述（填空答案）**：

> **每个事务都遵循两段锁协议**  
> 或 **所有事务都满足两段锁协议**

## 第7题

> 事务在内部需要被锁定的原因是 **保证事务的隔离性**。

**解析**：

- 多个事务并发执行时，若不加以控制，会出现：  
  丢失更新、脏读、不可重复读、幻读等问题。
- 通过**加锁**可以控制并发事务对数据的访问顺序，从而**实现隔离性**。
- 隔离性是 ACID 之一，锁是实现隔离性的主要手段。

**完整表述（填空答案）**：

> **保证事务的隔离性**  
> 或 **防止并发访问导致的数据不一致问题**

### 1. 请简述数据库、数据库管理系统的概念及数据库系统与两者的关系

**答：**

- **数据库（DB）**：长期存储在计算机内、有组织、可共享的大量数据的集合。
- **数据库管理系统（DBMS）**：位于用户与操作系统之间的数据管理软件，负责对数据库进行统一管理和控制。
- **数据库系统（DBS）**：由数据库、数据库管理系统、应用程序和数据库管理员组成的整体。

**关系**：  
数据库系统 = 数据库 + 数据库管理系统 + 应用程序 + 数据库管理员  
即 **DBS 包含 DB 和 DBMS**，DBMS 是管理 DB 的软件。

---

### 2. 请简述数据库管理系统关于调度的两段锁协议及其效果

**答：**

**两段锁协议（2PL）**：  
要求每个事务将加锁阶段和解锁阶段严格分开，即：

- **增长阶段**：只能加锁，不能释放锁
- **收缩阶段**：只能释放锁，不能加新锁

**效果**：

- 若所有事务都遵循两段锁协议，则任何调度都是**冲突可串行化**的。
- 但 **2PL 不能避免死锁**，需要额外的死锁检测或预防机制。

---

### 3. 什么是可串行化调度？什么是冲突可串行化调度？

**答：**

- **可串行化调度**：并发调度的执行结果与**某个串行调度**的结果相同，则称该调度是可串行化的。
- **冲突可串行化调度**：若调度可以通过交换**非冲突操作**转换为某个串行调度，则称该调度是冲突可串行化的。

**关系**：  
冲突可串行化调度一定是可串行化调度，反之不一定。

---

### 4. 请简答数据库管理系统中事务的四个特性

**答：**

| 特性                   | 含义                |
| -------------------- | ----------------- |
| **原子性（Atomicity）**   | 事务中的操作要么全做，要么全不做  |
| **一致性（Consistency）** | 事务执行前后，数据库保持完整性约束 |
| **隔离性（Isolation）**   | 并发事务之间互不干扰        |
| **持久性（Durability）**  | 事务提交后，修改永久保存      |

> 记忆口诀：**A CID**

---

### 5. 数据库系统中关系模型有哪几部分组成？其中约束有哪几种？试列出。

**答：**

**关系模型的三要素**：

1. 数据结构（二维表）
2. 数据操作（关系代数、关系演算）
3. 完整性约束

**完整性约束的三种类型**：

- **实体完整性**：主键不能为空
- **参照完整性**：外键要么为空，要么引用存在的候选码
- **用户定义完整性**：如 `CHECK(age >= 16)`

---

### 6. ER→关系模型转化时需要考虑哪些约束？简单解释 1NF, 2NF, 3NF 的概念

**答：**

**ER→关系模型转化的约束**：

- 主键、外键约束
- 实体完整性、参照完整性、用户定义完整性

**范式概念**：

| 范式      | 定义                         |
| ------- | -------------------------- |
| **1NF** | 属性值不可再分（原子性）               |
| **2NF** | 满足1NF，且非主属性完全依赖于主键（无部分依赖）  |
| **3NF** | 满足2NF，且非主属性不传递依赖于主键（无传递依赖） |

> 简记：1NF 无重复列，2NF 无部分依赖，3NF 无传递依赖

## 一、数据库设计中包含哪三种设计？

数据库设计通常分为以下**三个阶段（三种设计）**：

| 设计阶段          | 主要任务                       | 产出        |
| ------------- | -------------------------- | --------- |
| **1. 概念结构设计** | 对用户需求进行抽象，构建独立于具体DBMS的概念模型 | E-R图      |
| **2. 逻辑结构设计** | 将概念模型转换为特定数据模型（如关系模型）      | 关系模式、视图   |
| **3. 物理结构设计** | 确定数据库在存储介质上的存储结构和存取方法      | 索引、分区、内模式 |

> 记忆点：**概念 → 逻辑 → 物理**，独立性依次降低

---

## 二、关系模型设计中最容易出现的两个陷阱

在 E-R 图向关系模型转换（或概念设计）过程中，最容易出现的是两个**建模陷阱**：

| 陷阱名称     | 产生原因                                                       | 后果                      | 解决方法                     |
| -------- | ---------------------------------------------------------- | ----------------------- | ------------------------ |
| **扇形陷阱** | 两个实体之间存在 **1:N** 联系，但这两个实体又**分别与另一个实体存在 1:N 联系**，导致路径语义不唯一 | 查询结果出现**重复计数**或**信息冗余** | 调整实体间联系次序，或**添加缺失的直接联系** |
| **深坑陷阱** | 一对多联系链中存在**可选参与**（部分参与），导致某些实体在查询中被“吞掉”或丢失                 | **信息丢失**，某些实体查询不到       | 在可能丢失信息的实体之间**添加直接联系**   |

---

### 简单示例：

**扇形陷阱**：  

- 部门(1) → 员工(N)  
- 员工(1) → 项目(N)  
  → 查询“项目属于哪个部门”可能产生歧义或重复

**深坑陷阱**：  

- 学校(1) → 学生(N)（所有学生都有学校）  
- 学生(1) → 获奖(N)（部分学生有获奖）  
  → 如果只通过“获奖”往回查“学校”，可能会丢失没有获奖的学生所在的学校信息

## **索引**是数据库中**用于加速数据检索的一种数据结构**

## 1. B+树索引（默认最常用）

**适用场景**  

- 等值查询  
- 范围查询（`>`、`<`、`BETWEEN`）  
- 排序（`ORDER BY`）  
- 前缀匹配（`LIKE 'abc%'`）

**举例**  

```sql
-- 范围查询
SELECT * FROM order WHERE amount BETWEEN 100 AND 500;

-- 排序
SELECT * FROM user ORDER BY register_time;
```

---

## 2. 哈希索引

**适用场景**  

- 精确等值查询（`=`、`IN`）  
- **不支持范围、不支持排序**

**举例**  

```sql
-- 等值查询
SELECT * FROM user WHERE phone = '13800001111';
```

> 典型应用：Redis、MySQL Memory 引擎、某些等值查询场景

---

## 3. 位图索引

**适用场景**  

- **低基数（取值个数很少）** 的列  
- 例如：性别、婚姻状态、订单状态、部门类型  
- 特别适合 **多个条件组合查询**（AND / OR）

**举例**  

```sql
-- 低基数列组合查询
SELECT * FROM employee 
WHERE gender = '男' AND marital_status = '已婚';
```

> 不适合：频繁更新（位图锁开销大）

---

## 4. 唯一索引 / 主键索引

**适用场景**  

- 需要保证**字段值唯一**  
- 常用于：身份证号、学号、邮箱、用户名

**举例**  

```sql
CREATE UNIQUE INDEX idx_email ON user(email);
```

---

## 5. 复合索引（联合索引）

**适用场景**  

- **多个条件同时查询**  
- **遵循最左前缀原则**

**举例**  

```sql
-- 索引 (a, b, c)
WHERE a = 1         ✅
WHERE a = 1 AND b = 2 ✅
WHERE a = 1 AND b = 2 AND c = 3 ✅
WHERE b = 2         ❌（不满足最左前缀）
```

> 经典考题：`(name, age)` 能支持哪些查询？

---

## 6. 全文索引

**适用场景**  

- **文本内容中的关键词搜索**  
- 如：文章、商品描述、评论

**举例**  

```sql
SELECT * FROM article 
WHERE MATCH(title, content) AGAINST('数据库 索引');
```

> 不是 `LIKE '%xxx%'`（那个不走索引）

---

## 7. 空间索引（R-tree）

**适用场景**  

- 地理位置 / 几何数据  
- 如：附近的人、外卖配送范围、GIS 系统

**举例**  

```sql
SELECT * FROM store 
WHERE ST_Distance(location, 'POINT(120 30)') < 1000;
```

---

## 8. 覆盖索引（不是独立类型，是一种策略）

**适用场景**  

- **查询的所有字段都在索引中**  
- 避免“回表”，性能最高

**举例**  

```sql
-- 索引 (id, name)
SELECT id, name FROM user WHERE id = 5;
```

---

| 索引类型   | 核心场景          | 反例 / 不适合 |
| ------ | ------------- | -------- |
| B+树    | 范围、排序、等值      | 无        |
| 哈希     | 精确等值          | 范围、排序    |
| **位图** | **低基数、固定几个值** | 频繁更新     |
| 唯一索引   | 唯一性约束         | 允许重复值    |
| 复合索引   | 多条件 + 最左前缀    | 跳过前导列    |
| 全文索引   | 文本关键词         | 精确匹配     |
| 空间索引   | 地理位置          | 普通字段     |

## 一、访问控制的定义

> **访问控制**：数据库管理系统（DBMS）对用户访问数据库资源的**权限管理机制**，防止未授权访问和数据泄露。

---

## 二、两种权限类型

| 权限类型     | 说明                  | 示例                                           |
| -------- | ------------------- | -------------------------------------------- |
| **系统权限** | 用户对数据库**系统级别**的操作权限 | `CREATE TABLE`、`CREATE USER`、`DROP DATABASE` |
| **对象权限** | 用户对**特定数据库对象**的操作权限 | `SELECT`、`INSERT`、`UPDATE`、`DELETE` 某个表      |

> 记忆：系统权限管**能不能建表**，对象权限管**能不能查某张表**

---

## 三、SQL 中的授权与收回（常考 SQL）

### 1. GRANT（授权）

```sql
-- 对象权限：授予用户对表的查询权限
GRANT SELECT ON Student TO user1;

-- 授予多个权限
GRANT SELECT, INSERT ON Student TO user2;

-- 允许用户把权限再转授给别人
GRANT SELECT ON Student TO user3 WITH GRANT OPTION;

-- 授予系统权限（语法因DBMS而异）
GRANT CREATE TABLE TO user4;
```

### 2. REVOKE（收回权限）

```sql
-- 收回查询权限
REVOKE SELECT ON Student FROM user1;

-- 级联收回（收回user2权限时，被user2授权的用户也会失去权限）
REVOKE SELECT ON Student FROM user2 CASCADE;
```

---

## 四、角色（Role）

> **角色**是权限的集合，可以**批量授权**给多个用户。

| 操作      | SQL 示例                                     |
| ------- | ------------------------------------------ |
| 创建角色    | `CREATE ROLE student_role;`                |
| 给角色授权   | `GRANT SELECT ON Student TO student_role;` |
| 把角色授予用户 | `GRANT student_role TO user1;`             |

**优点**：简化权限管理，提高维护效率。

---

## 五、访问控制的实现方式

| 方式                  | 说明                      | 粒度    |
| ------------------- | ----------------------- | ----- |
| **自主访问控制（DAC）**     | 用户可自主将自己的权限授予他人         | 表级、列级 |
| **强制访问控制（MAC）**     | 系统根据安全级别（密级）控制访问，用户无法绕过 | 行级、列级 |
| **基于角色的访问控制（RBAC）** | 权限赋予角色，角色赋予用户           | 灵活    |

> 大多数商用数据库（如 Oracle、MySQL）主要使用 **DAC + RBAC**

---

## 六、考试常见题型

### 1. 选择题

**例**：以下哪个命令用于授予对象权限？  
A. `REVOKE`  
B. `GRANT`  
C. `CREATE`  
D. `DENY`  
**答案**：B

### 2. 填空题

**例**：数据库中的权限分为 **系统权限** 和 **______**。  
**答案**：对象权限

### 3. SQL 写语句

**例**：将 Student 表的 SELECT 权限授予用户 'Tom'，并允许 Tom 将该权限转授给其他用户。  
**答案**：

```sql
GRANT SELECT ON Student TO Tom WITH GRANT OPTION;
```

### 4. 简答题

**例**：简述角色（Role）在数据库访问控制中的作用。  
**答案**：角色是权限的集合，将权限先授予角色，再将角色授予用户，可以简化权限管理，提高维护效率，尤其适用于用户数量多且权限相同的场景。

## 七、速记

> **GRANT 授权，REVOKE 收回，WITH GRANT OPTION 可转授，角色=权限集合，RBAC 最常用。**

# 四、数据库设计

## 1、数据库系统生存周期模型

**定义**：  
数据库系统生存周期是指从数据库**规划、设计、实现、运行维护**到**最终淘汰**的整个过程。

**主要阶段**（按顺序）：

| 阶段    | 说明                   |
| ----- | -------------------- |
| 数据库规划 | 明确任务目标和范围            |
| 需求分析  | 收集用户需求，输出需求规格说明书     |
| 概念设计  | 构建独立于DBMS的概念模型（E-R图） |
| 逻辑设计  | 转换为特定数据模型（如关系模式）     |
| 物理设计  | 确定存储结构、索引、存取方法       |
| 实现    | 建表、编写应用、数据加载         |
| 运行与维护 | 日常监控、性能调优、备份恢复       |

---

## 2、数据库概念设计：ER建模技术、方法

**定义**：  
概念设计是将用户需求抽象为**独立于任何DBMS的概念模型**，常用**E-R模型**。

**E-R模型三要素**：

| 要素  | 图形  | 说明                   |
| --- | --- | -------------------- |
| 实体  | 矩形  | 现实世界中可独立存在的对象        |
| 属性  | 椭圆  | 实体的特征（单值/多值/派生/复合）   |
| 联系  | 菱形  | 实体之间的关联（1:1、1:N、M:N） |

**E-R建模方法**：

1. 识别实体类型
2. 识别联系类型
3. 识别并关联属性
4. 确定属性域
5. 确定主键
6. 检查冗余
7. 验证与用户确认

---

## 3、数据库逻辑设计：规范化理论、ER模型映射为关系模型

**定义**：  
将概念模型（E-R图）转换为**特定数据模型**（如关系模型）的过程。

### 规范化理论（范式）

| 范式       | 定义                             |
| -------- | ------------------------------ |
| **1NF**  | 属性值不可再分（原子性）                   |
| **2NF**  | 满足1NF + 非主属性**完全依赖**于主键（无部分依赖） |
| **3NF**  | 满足2NF + 非主属性**不传递依赖**于主键       |
| **BCNF** | 所有**决定因素都是候选码**（函数依赖范围内彻底分解）   |

### E-R映射为关系模型的核心规则

| E-R元素  | 映射方法                |
| ------ | ------------------- |
| 强实体    | 一个表，主键=实体主键         |
| 弱实体    | 一个表，主键=弱实体主键+所属实体主键 |
| 1:N 联系 | N方表中加1方的主键作为外键      |
| 1:1 联系 | 合并为一张表，或一方主键放入另一方   |
| M:N 联系 | 新建关系表，包含双方主键（组合主键）  |
| 多值属性   | 新建表，包含实体主键+多值属性     |

# 五、事务管理与并发控制

## 1、事务概念

**定义**：  
事务是数据库操作的**逻辑工作单元**，是一组要么全部执行、要么全部不执行的SQL语句序列。

---

## 2、事务的ACID特性

| 特性  | 含义         | 实现机制            |
| --- | ---------- | --------------- |
| 原子性 | 全做或全不做     | 恢复协议（Undo）      |
| 一致性 | 事务前后数据保持完整 | 完整性约束、触发器       |
| 隔离性 | 并发事务互不干扰   | 并发控制协议（如两段锁）    |
| 持久性 | 提交后修改永久保存  | 恢复协议（Redo + 日志） |

---

## 3、并发控制的必要性和基本思想

**必要性**：  
多事务并发执行可能导致：

- 丢失更新
- 脏读
- 不可重复读
- 幻读

**基本思想**：  
通过加锁、时间戳、多版本并发控制（MVCC）**等机制，保证事务的**隔离性，使并发调度结果与某个串行调度结果一致。

---

## 4、调度基本概念、两段锁协议、严格两段锁协议、冲突可串行化调度

| 术语         | 定义                       |
| ---------- | ------------------------ |
| **调度**     | 多个并发事务操作的执行顺序序列          |
| **串行调度**   | 事务连续执行，无交叉               |
| **可串行化调度** | 并发调度结果等同于某个串行调度          |
| **冲突可串行化** | 通过交换非冲突操作可转换为串行调度（优先图无环） |

**两段锁协议（2PL）**：

- 增长阶段：只能加锁，不能释放
- 收缩阶段：只能释放，不能加锁
- 若所有事务遵循2PL → 调度是冲突可串行化的

**严格两段锁协议（Rigorous 2PL）**：

- 所有锁（尤其是专用锁）**直到事务提交/回滚后才释放**
- 优点：避免级联回滚，保证可恢复性

---

## 5、死锁的概念和解决办法

**定义**：  
两个或多个事务**相互等待对方持有的锁**，导致无法继续执行。

**解决办法**：

| 方法          | 说明                   |
| ----------- | -------------------- |
| **死锁预防**    | 一次性申请所有锁、按固定顺序加锁     |
| **死锁检测与解除** | 构造**等待图**，发现环则选择事务回滚 |

---

## 6、数据库恢复、日志

**定义**：  
数据库恢复是将数据库从**故障状态恢复到一致状态**的过程。

**日志（Log）**：  
记录事务执行过程中**所有更新操作**的历史信息（旧值、新值）。

**WAL原则（预写日志）**：  
**先写日志，后写数据库**（保证原子性和持久性）。

**恢复操作**：

| 操作   | 作用         |
| ---- | ---------- |
| Undo | 撤销未提交事务的修改 |
| Redo | 重做已提交事务的修改 |

**检查点（Checkpoint）**：  
定期将内存修改刷入磁盘，**减少恢复时需要扫描的日志量**。

---

# 六、查询性能分析

**定义**：  
通过分析SQL语句的执行计划、索引使用、I/O开销等，找出性能瓶颈并进行优化。

**常见优化策略**：

- 使用索引（B+树、位图等）
- 避免全表扫描
- 优化连接顺序（先做选择，后做连接）
- 避免在WHERE子句中对字段进行函数运算

**启发式优化规则**：

1. 选择运算尽可能先做
2. 投影运算与选择同时进行
3. 选择与笛卡尔积结合为连接运算
4. 找出公共子表达式并复用

# 七、非结构化数据的表示与存储、大数据时代下的NoSQL技术

## 1、非结构化数据

**定义**：  
没有固定数据模型或格式的数据，如：文本、图片、音频、视频、XML、JSON等。

**存储方式**：

- 文件系统
- 对象存储
- NoSQL数据库（文档、键值、列族、图）

## 2、NoSQL技术

**定义**：Not Only SQL，非关系型数据库，适合海量数据、高并发、灵活Schema的场景。

**四大类型**：

| 类型    | 代表              | 特点               |
| ----- | --------------- | ---------------- |
| 键值数据库 | Redis           | 简单快速，适合缓存        |
| 列族数据库 | HBase、Cassandra | 适合大规模数据分析        |
| 文档数据库 | MongoDB         | 灵活Schema，类似JSON  |
| 图数据库  | Neo4j           | 适合关系密集型（社交网络、推荐） |

**与关系型数据库对比**：

| 对比项    | NoSQL      | 关系型    |
| ------ | ---------- | ------ |
| 数据模型   | 非结构化/半结构化  | 结构化表   |
| Schema | 灵活         | 固定     |
| 扩展方式   | 水平扩展       | 垂直扩展为主 |
| ACID   | 弱支持（最终一致性） | 强支持    |

---

## 3、XML（可扩展标记语言）

**定义**：  
一种用于**传输和存储数据**的标记语言，用户可自定义标签。

**作用**：

- 数据交换与共享
- 配置文件
- 存储半结构化数据

**良构 vs 有效**：

- **良构**：语法正确（一个根元素、标签关闭、正确嵌套）
- **有效**：良构 + 遵守DTD或XML Schema约束

## 二、关系模型特点

1. **数据结构单一**：使用二维表表示数据和数据之间的联系。
2. **基于集合论**：以关系代数、关系演算为理论基础。
3. **数据独立性高**：逻辑独立性和物理独立性较好。
4. **支持完整性约束**：实体完整性、参照完整性、用户定义完整性。
5. **非过程化**：用户只需说明“做什么”，不需说明“怎么做”。
6. **关系必须规范化**：基本要求是满足第一范式（1NF）。

| **原子性** | 全做或全不做     | 恢复协议（Undo）      |
| **一致性** | 事务前后数据保持完整 | 完整性约束、触发器       |
| **隔离性** | 并发事务互不干扰   | 并发控制协议（如两段锁）    |
| **持久性** | 提交后修改永久保存  | 恢复协议（Redo + 日志） |

## 六、SQL的特点

## 九、两段锁协议及其与冲突可串行化调度的关系

### 两段锁协议（2PL）

- **增长阶段**：只能加锁，不能释放锁
- **收缩阶段**：只能释放锁，不能加新锁

### 与冲突可串行化调度的关系

> **如果所有并发事务都遵循两段锁协议，则它们的任何调度都是冲突可串行化的。**

- 2PL 是保证冲突可串行化的**充分条件**（不是必要条件）。
- 2PL **不能避免死锁**。

## 十、并发控制与数据库恢复

| 概念        | 定义                 | 主要技术                  |
| --------- | ------------------ | --------------------- |
| **并发控制**  | 保证并发事务正确执行，防止数据不一致 | 两段锁（2PL）、MVCC、时间戳     |
| **数据库恢复** | 将数据库从故障状态恢复到一致状态   | 日志（Undo/Redo）、WAL、检查点 |

**并发控制必要性**：防止丢失更新、脏读、不可重复读、幻读。

**恢复核心操作**：

- **Undo**：撤销未提交事务的修改
- **Redo**：重做已提交事务的修改

**WAL原则**：先写日志，后写数据库。

## 十一、XML基本思想与特点

**基本思想**：  
可扩展标记语言，通过**自定义标签**来描述数据的**结构和含义**，实现**跨平台、跨系统的数据交换与存储**。

**特点**：

| 特点      | 说明                |
| ------- | ----------------- |
| 可扩展     | 标签没有预定义，用户可自定义    |
| 自我描述    | 标签名说明数据含义         |
| 纯文本     | 任何能处理文本的程序都能处理XML |
| 树形结构    | 有且只有一个根元素，形成文档树   |
| 与HTML互补 | XML传数据，HTML显示数据   |

## 十二、MongoDB基本思想与特点

**基本思想**：  
介于关系型和非关系型之间的**文档数据库**，以 **JSON / BSON** 格式存储数据，**模式自由**，支持水平扩展。

**特点**：

| 特点      | 说明                     |
| ------- | ---------------------- |
| 面向集合存储  | 数据分组存储在Collection（类似表） |
| 模式自由    | 同一集合可存储不同结构的文档         |
| 文档模型    | 每条记录是一个JSON/BSON文档     |
| 支持复制与分片 | 高可用 + 水平扩展             |
| 不支持多表连接 | 通过嵌套文档或多次查询实现          |

**术语对应**：

| 关系型         | MongoDB    |
| ----------- | ---------- |
| Table       | Collection |
| Row         | Document   |
| Column      | Field      |
| Primary Key | `_id`      |
