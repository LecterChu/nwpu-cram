第1章：数据库基础
=========

一、数据管理技术三阶段
-----------

| 阶段    | 年代          | 硬件           | 软件          | 数据特点             |
| ----- | ----------- | ------------ | ----------- | ---------------- |
| 手工管理  | 1950s前      | 打孔卡片、磁带（无磁盘） | 无OS，无数据管理软件 | 程序与数据相互依赖，无共享    |
| 文件系统  | 1950s-1960s | 磁盘、磁鼓        | OS + 文件系统   | 文件相互独立，共享困难，冗余难免 |
| 数据库系统 | 1960s-      | 大容量磁盘        | DBMS        | 数据彻底共享，透明访问      |

二、核心概念定义
--------

| 术语                | 定义                                                                     |
| ----------------- | ---------------------------------------------------------------------- |
| **数据**            | 描述事物的符号记录                                  |
| **信息**            | 经过加工处理、有含义的数据                                                          |
| **数据库(DB)**       | 大量、有组织、结构化、长期存储、可共享的数据集合                                               |
| **数据库管理系统(DBMS)** | 位于用户与数据库之间的管理软件，负责管理数据和提供访问接口。两个基本功能：一是负责对底层数据库中的数据管理；二是给用户提供访问数据库的接口。 |
| **数据库系统(DBS)**    | DB（数据库） + DBMS（数据库管理系统） + 应用程序                                         |

**常见DBMS举例**：Oracle, DB2, SQL Server, PostgreSQL, MySQL, OpenGauss, OceanBase

----------------

三、数据库作为数据管理方法的特点
----------------

1. **数据抽象**：数据统一结构化，结构信息独立于数据

2. **高可靠**：支持完整性约束，保证数据一致性

3. **高效**：物理存储优化、控制冗余、查询优化、并发控制

四、数据模型（核心）
----------

### 1. 什么是数据模型

> 对客观世界事物的**数据模拟与抽象**，通过数据、数据之间的联系、约束来表达事物特征。

### 什么是模型

> 客观世界事物的模拟与抽象。

| 要素        | 说明                |

### 3. 数据模型建模三原则

1. 能比较真实反映客观事物

2. 容易理解（便于交流）

3. 容易计算机实现

第2-2讲：关系模型
==========

### 2. 关系模型的数据结构

* 关系 = 数学概念，物理上表示为**二维表（Table）**

* 关系的数学定义：
  
  * 笛卡尔积 D1​×D2​×⋯×Dn​ 的任意子集称为关系

### 3. 关系的性质（重点）

| 性质     | 说明                |
| ------ | ----------------- |
| 有限集合   | 关系不能无限            |
| 属性顺序无关 | 列的顺序可以交换          |
| 属性值同质  | 一列中数据类型相同         |
| 元组顺序无关 | 行的顺序可以交换          |
| 原子值    | 每个单元格只能有一个值（不可再分） |
| 无重复元组  | 不允许两行完全相同         |

### 4. 关系中的键（码）

| 类型  | 定义         | 例子               |
| --- | ---------- | ---------------- |
| 超码  | 唯一标识元组的属性集 | (学号, 姓名)         |
| 候选码 | 最小超码       | (学号)             |
| 主码  | 被选中的候选码    | 学号               |
| 辅码  | 未被选中的候选码   | 身份证号             |
| 外码  | 引用其他关系的候选码 | 学生表中的“班级编号”引用班级表 |

### 6. 完整性约束（重点）

| 完整性     | 含义                   | 示例                |
| ------- | -------------------- | ----------------- |
| 实体完整性   | 主码不能为空               | 学号不能为空            |
| 参照完整性   | 外码值要么为空，要么等于被参照表的主码值 | 学生表中的班级编号必须存在于班级表 |
| 用户定义完整性 | 用户自定义规则              | `age >= 16`       |

第2-3讲：关系代数
===============

一、关系代数概述
--------

### 1. 什么是关系代数

* 一种**过程化查询语言**

* 操作对象是**关系（表）**

* 操作结果也是**关系（表）**

> 核心特征：**封闭性**（输入输出都是关系）

### 2. 关系代数 vs 关系演算

| 对比项 | 关系代数          | 关系演算          |
| --- | ------------- | ------------- |
| 关注点 | **怎么做**       | **要什么**       |
| 类型  | 过程化           | 声明式（非过程化）     |
| 基础  | 集合运算          | 谓词逻辑          |
| 例子  | σ, Π, ∪, −, ⨝ | { t \| F(t) } |

* * *

二、基本操作（必须掌握）
------------

### 1. 并（Union）∪

* 要求：两个关系**并兼容**（属性个数相同，对应域相同）

* 结果：R ∪ S = {t | t ∈ R ∨ t ∈ S}

* 去重

### 2. 差（Difference）−

* 要求：并兼容

* 结果：R − S = {t | t ∈ R ∧ t ∉ S}

### 3. 交（Intersection）∩

* 要求：并兼容

* 结果：R ∩ S = {t | t ∈ R ∧ t ∈ S}

* 可表示为：R ∩ S = R − (R − S)

### 4. 笛卡尔积（Cartesian Product）×

* 结果属性数 = m + n

* 结果元组数 = |R| × |S|

* 所有组合

* * *

三、专门操作（重点）
----------

### 1. 选择（Selection）σ

* 按**行**筛选

* 格式：σ<sub>条件</sub>(R)

* 条件可包含：=, ≠, <, ≤, >, ≥, ∧, ∨, ¬

### 2. 投影（Projection）Π

* 按**列**筛选

* 格式：Π<sub>属性列表</sub>(R)

* **自动去重**

### 3. θ-连接（Theta Join）⨝<sub>θ</sub>

* 格式：R ⨝<sub>θ</sub> S = σ<sub>θ</sub>(R × S)

* θ是比较运算符（=, <, >, ≠ 等）

### 4. 相等连接（Equijoin）

* θ = = 的特殊情况

* 结果中包含两个连接属性（保留重复列）

### 5. 自然连接（Natural Join）⭐ 最重要

* 格式：R ⨝ S

* 自动在**所有同名属性**上做相等连接

* **只保留一个连接列**（去重）

### 6. 外连接（Outer Join）

* **左外连接** ⟕：保留左表中不匹配的元组，右表部分补 NULL

* **右外连接** ⟖：保留右表中不匹配的元组

* **全外连接** ⟗：两边都保留

四、除法（Division）÷ ⭐ 最难
--------------------

### 1. 除法定义

给定关系R(X,Y)和S(Y,Z)，其中X, Y, Z为属性组。 R中的Y与S中的Y可以有不同的属性名，但必须出自相同的域集。R与S的除运算得到一个新的关系P(X)，P是R中满足下列条件的元组在X属性列上的投影：元组在X上分量值x的象集Yx包含S在Y上投影的集合。记作:

$R ÷ S = \{ t[X]∣t∈R∧Π_Y​(S)⊆Yx \}$

### 一、关系代数 vs 关系演算

| 对比项  | 关系代数        | 关系演算        |

第3讲：SQL简介
=========
一、SQL是什么

--------

| 项目  | 内容                                 |
| --- | ---------------------------------- |
| 全称  | Structured Query Language（结构化查询语言） |
| 中文  | 结构化查询语言                            |
| 作用  | 访问和操作数据库的标准语言                      |
| 特点  | 非过程化、基于关系代数和关系演算、易学                |

二、SQL的目标（设计目的）
--------------

1. 创建数据库和关系结构（表）

2. 执行基本的数据管理任务（增删改查）

3. 执行简单和复杂的查询

4. **以最小的用户工作量完成上述任务**

> 第4点体现了SQL的**非过程化**特点：用户只需说“要什么”，不用管“怎么做”。

三、SQL的功能分类（考试重点 ⭐）
------------------

| 功能       | 动词                     | 说明          |
| -------- | ---------------------- | ----------- |
| **数据查询** | SELECT                 | 查询数据        |
| **数据定义** | CREATE, DROP, ALTER    | 定义/删除/修改表结构 |
| **数据操纵** | INSERT, UPDATE, DELETE | 插入、更新、删除数据  |
| **数据控制** | GRANT, REVOKE          | 授权、收回权限     |

四、SQL的使用方式
----------

| 方式         | 说明                    | 特点          |
| ---------- | --------------------- | ----------- |
| **交互式SQL** | 在终端直接输入SQL语句执行        | 即输即得，适合临时查询 |
| **嵌入式SQL** | 将SQL嵌入到高级语言（C、Java等）中 | 适合写应用程序     |

五、SQL的特点
--------

1. **非过程化**：只描述要什么，不描述怎么获取

2. **基于关系代数和关系演算**：理论基础扎实

3. **易学**：接近自然语言

4. **一体化**：集数据定义、操纵、查询、控制于一体

5. **两种使用方式**：交互式 + 嵌入式

二、数据定义语言（DDL）
-------------

### 1. CREATE TABLE（建表）

```sql
CREATE TABLE Student (
    sNo   CHAR(8) PRIMARY KEY,
    sName VARCHAR(20) NOT NULL,
    sex   CHAR(1),
    age   INT,
    dNo   CHAR(3),
    FOREIGN KEY (dNo) REFERENCES Department(dNo)
);
```

**完整性约束**：

* 实体完整性：`PRIMARY KEY`

* 参照完整性：`FOREIGN KEY ... REFERENCES`

* 用户定义：`NOT NULL`, `UNIQUE`, `CHECK`, `DEFAULT`

### 2. ALTER TABLE（修改表）

```sql
ALTER TABLE Student ADD email VARCHAR(50);
ALTER TABLE Student DROP COLUMN email;
```

### 3. DROP TABLE（删除表）

```sql
DROP TABLE Student;
-- RESTRICT: 有依赖则拒绝删除
-- CASCADE: 级联删除依赖对象
```

三、数据查询语言（DQL）—— SELECT ⭐⭐⭐
--------------------------

### SELECT 完整语法顺序（必背）

```sql
SELECT [DISTINCT] 列名  (DISTINCT：去重)
FROM 表名
[WHERE 条件]
[GROUP BY 分组列]
[HAVING 分组条件]
[ORDER BY 排序列 [ASC|DESC]];
```

### 2. WHERE 条件谓词

| 谓词类型 | 运算符/关键字                              |
| ---- | ------------------------------------ |
| 比较   | `=`, `>`, `<`, `>=`, `<=`, `<>`      |
| 范围   | `BETWEEN ... AND ...`, `NOT BETWEEN` |
| 集合   | `IN`, `NOT IN`                       |
| 模式匹配 | `LIKE`, `NOT LIKE`                   |
| 空值   | `IS NULL`, `IS NOT NULL`             |
| 逻辑   | `AND`, `OR`, `NOT`                   |

**LIKE 通配符**：

* `%`：0个或多个字符

* `_`：单个字符

* 转义：`ESCAPE '\'`

### 4. ORDER BY（排序）

```sql
SELECT * FROM Student ORDER BY dNo, age DESC;
-- NULL 在 ORDER BY 中：默认视为最大或最小（取决于DBMS
-- DESC降序，ASC升序
```

### 5. 聚合函数

### （⭐只能用在 SELECT 和 HAVING，不能用于WHERE）

| 函数                  | 说明      |
| ------------------- | ------- |
| `COUNT(*)`          | 行数      |
| `COUNT(DISTINCT 列)` | 非空不同值个数 |
| `SUM`               | 和（数值）   |
| `AVG`               | 平均（数值）  |
| `MAX`               | 最大      |
| `MIN`               | 最小      |

**注意**：除 `COUNT(*)` 外，其他函数**自动忽略 NULL**。

### 6. GROUP BY 分组

规则：SELECT 中的非聚合列**必须**出现在 GROUP BY 中。

```sql
SELECT cNo, score, COUNT(sNo) FROM SC GROUP BY cNo, score;
```

### 7. HAVING 分组后筛选

```sql
-- 选课超过3门的学生
SELECT sNo FROM SC GROUP BY sNo HAVING COUNT(*) > 3;
```

> WHERE 在分组前，HAVING 在分组后。

四、子查询（Subquery）
---------------

### 1. 子查询类型

| 维度              | 标量子查询       | 行子查询         | 表子查询                  |
| --------------- | ----------- | ------------ | --------------------- |
| **返回**          | 单行单列        | 单行多列         | 多行多列                  |
| **内存形状**        | 单个值         | 一条记录         | 一张表                   |
| **放在 WHERE 里**  | 用 `=`、`>` 等 | 用 `=` 配合多字段  | 用 `IN`、`EXISTS`、`ANY` |
| **放在 SELECT 里** | ✅ 可以        | ❌ 不行         | ❌ 不行                  |
| **放在 FROM 里**   | ❌ 不行        | ❌ 不行（需加括号变表） | ✅ 可以（必须加别名）           |

放在 SELECT 里
-----------

```sql
-- ✅ 正确：标量子查询返回一个数字
SELECT name, (SELECT AVG(age) FROM Student) AS avg_age FROM Student;

-- ❌ 错误：行子查询返回 (cNo, age) 两列，放不进一个字段
SELECT name, (SELECT cNo, age FROM Student WHERE name = '张三') FROM Student;

-- ❌ 错误：表子查询返回多行，更不行
SELECT name, (SELECT * FROM Student) FROM Student;
```

放在 FROM 里
---------

```sql
-- ❌ 错误：标量子查询返回数字 20，FROM 后面不能跟数字
SELECT * FROM (SELECT AVG(age) FROM Student);

-- ✅ 正确：表子查询返回多行多列，但必须加别名
SELECT * FROM (SELECT * FROM Student WHERE age > 18) AS t;

-- 注意：行子查询如果强行放在 FROM 里，实际上就变成了表子查询（因为加了括号）
SELECT * FROM (SELECT cNo, age FROM Student WHERE name = '张三') AS t;  -- 这是合法的，但本质已经是表子查询
```

### 2. 子查询中的 NULL 陷阱

```sql
-- ❌ 错误：子查询包含 NULL，NOT IN 返回空
SELECT dName FROM Department 
WHERE dNo NOT IN (SELECT dNo FROM Student WHERE age<18);

-- ✅ 正确：排除 NULL
SELECT dName FROM Department 
WHERE dNo NOT IN (SELECT dNo FROM Student WHERE age<18 AND dNo IS NOT NULL);
```

### 3. ANY / ALL

| 条件      | 等价    |
| ------- | ----- |
| `= ANY` | IN    |
| `> ANY` | > MIN |
| `< ANY` | < MAX |
| `> ALL` | > MAX |
| `< ALL` | < MIN |

### 4. 相关子查询

```sql
-- 没选01号课程的学生
SELECT sName FROM Student S
WHERE NOT EXISTS (
    SELECT * FROM SC WHERE sNo=S.sNo AND cNo='01'
);
```

### 5. EXISTS / NOT EXISTS 实现全称量词

**全称量词转换**：∀x P ≡ ¬∃x (¬P)

```sql
-- 选修了全部课程的学生 = 不存在一门课程他没选
SELECT sName FROM Student S
WHERE NOT EXISTS (
        --选出不存在一门没选的课的学生
    SELECT * FROM Course C
    WHERE NOT EXISTS (
        --选出当前学生没选的课程
        SELECT * FROM SC WHERE sNo=S.sNo AND cNo=C.cNo
        --选出当前学生选修的课程
    )
);
```

建议用not exists代替not in :

| 维度             | NOT EXISTS          | NOT IN                |
| -------------- | ------------------- | --------------------- |
| **对 NULL 的处理** | ✅ 安全（只判断有无行）        | ❌ 危险（子查询有 NULL 则结果全空） |
| **子查询返回结果集**   | 不关心具体值，只关心是否有行      | 必须返回一个值列表（不能有多列）      |
| **关联性**        | 通常是**关联子查询**（引用外层列） | 通常是**非关联子查询**（独立执行）   |
| **执行逻辑**       | 逐行判断外层表             | 先执行子查询，再整体判断          |
| **性能（通常）**     | 找到第一个满足条件即可停止       | 必须完成全部子查询             |

五、多表查询（JOIN）
------------

### 1. 等值连接（传统写法）

```sql
SELECT s.sName, c.cName, sc.score
FROM Student s, Course c, SC sc
WHERE s.sNo=sc.sNo AND c.cNo=sc.cNo;
```

### 2. JOIN 语法

```sql
-- 自然连接
SELECT * FROM Student NATURAL JOIN SC;
-- 内连接 USING
SELECT * FROM SC JOIN Course USING(cNo);
-- 内连接 ON
SELECT * FROM SC JOIN Course ON SC.cNo=Course.cNo;
-- 左外连接（保留左表所有行）
SELECT * FROM Student LEFT OUTER JOIN SC ON Student.sNo=SC.sNo;
-- 右外连接、全外连接类似
```

### 3. 自连接

```sql
-- 查询数据库课程的先修课程名
SELECT c2.cName
FROM Course c1, Course c2
WHERE c1.cPNo=c2.cNo AND c1.cName='数据库';
```

第3-2讲：高级SQL
================

一、索引（Index）
-----------

### 1. 什么是索引

* 一种**加速访问**数据库表的数据结构

* 基于一列或多列的值创建

* 以**空间换时间**（占用额外存储，提升查询速度）

### 2. 创建和删除索引

```sql
-- 创建索引
CREATE INDEX idx_sName ON Student(sName);
-- 创建唯一索引（不允许重复值）
CREATE UNIQUE INDEX idx_sNo ON Student(sNo);
-- 删除索引
DROP INDEX idx_sName;
```

### 3. 索引的优缺点

| 优点                           | 缺点               |
| ---------------------------- | ---------------- |
| 加速查询（特别是WHERE、ORDER BY、JOIN） | 占用额外存储空间         |
| 保证唯一性（UNIQUE INDEX）          | 增删改操作变慢（需同步更新索引） |

* * *

二、视图（View）
-------------

### 1. 什么是视图

* **虚拟表**，不实际存储数据

* 存储的是**查询定义**，数据来自基表

* 每次查询视图时，DBMS执行视图定义查询

> 记忆点：**视图 = 存储的SELECT语句**

### 2. 创建和删除视图

```sql
-- 创建视图
CREATE VIEW view_1 AS
SELECT s.sName, c.cName, sc.score
FROM Student s, Course c, SC sc
WHERE s.sNo=sc.sNo AND c.cNo=sc.cNo AND c.cName='离散数学';

-- 带列名创建
CREATE VIEW view_2(cNo, stuCount) AS
SELECT cNo, COUNT(*) FROM SC GROUP BY cNo;

-- 删除视图
DROP VIEW view_1;
DROP VIEW view_2 CASCADE;  -- 级联删除依赖对象
```

### 3. WITH CHECK OPTION（重要）

作用：**禁止插入/更新那些会导致行从视图中消失的数据**

```sql
CREATE VIEW IS_Student AS
SELECT * FROM Student 
WHERE dNo='01' 
WITH CHECK OPTION;
-- ❌ 下面这条插入会失败，因为dNo='02'不在视图中
INSERT INTO IS_Student VALUES('999', '张三', '男', 20, '02');
```

### 4. 视图解析（View Resolution）

DBMS会将视图查询**合并**到用户查询中执行：

```sql
-- 用户查询
SELECT sName, score FROM view_1 WHERE score<60;
-- 实际执行（等价展开）
SELECT sName, sc.score 
FROM Student s, Course c, SC sc
WHERE s.sNo=sc.sNo AND c.cNo=sc.cNo 
  AND c.cName='离散数学' AND sc.score<60; --sc.score<60 被合并到视图中
```

### 5. 视图的可更新性（Updatability）

**不是所有视图都能更新**。满足以下条件才能更新：

| 条件        | 说明              |
| --------- | --------------- |
| 单表        | 视图基于单个基表        |
| 包含主键      | 视图包含基表的主键       |
| 无聚合       | 没有GROUP BY、聚合函数 |
| 无DISTINCT | 没有DISTINCT      |
| 无子查询      | 没有子查询（部分允许）     |

**不可更新的视图类型**：

* 多表连接视图

* 包含GROUP BY、聚合函数

* 包含DISTINCT

* 包含子查询（部分）

### 6. 视图的优缺点

| 优点         | 缺点     |
| ---------- | ------ |
| 数据独立性      | 更新受限   |
| 安全性（隐藏敏感列） | 结构受限   |
| 简化复杂查询     | 性能可能下降 |
| 方便定制       | —      |

三、事务（Transaction）⭐⭐⭐
--------------------

### 1. 什么是事务

> **一组数据库操作的逻辑单元**，要么全部成功，要么全部失败（原子性）

### 2. 事务的生命周期

```sql
-- 显式开启事务
BEGIN TRANSACTION;
UPDATE account SET balance = balance - 10 WHERE id = '01';
UPDATE account SET balance = balance + 10 WHERE id = '02';
-- 提交（永久生效）
COMMIT;
-- 或回滚（撤销所有修改）
ROLLBACK;
```

### 3. 事务的ACID特性

| 特性      | 含义               |
| ------- | ---------------- |
| **原子性** | 事务中的操作要么全做，要么全不做 |
| **一致性** | 事务执行前后，数据库状态一致   |
| **隔离性** | 多个事务并发执行互不干扰     |
| **持久性** | 事务提交后，修改永久保存     |

### 4. 事务边界

| 方式  | 说明                       |
| --- | ------------------------ |
| 隐式  | 每个SQL语句自动作为事务（部分DBMS）    |
| 显式  | 用BEGIN/COMMIT/ROLLBACK控制 |

四、完整性约束（Integrity Constraints）
------------------------------

### 1. 约束类型

| 类型    | 关键词                              |
| ----- | -------------------------------- |
| 实体完整性 | PRIMARY KEY                      |
| 参照完整性 | FOREIGN KEY                      |
| 用户定义  | CHECK, NOT NULL, UNIQUE, DEFAULT |

实体完整性 —— `PRIMARY KEY` (主键)

参照完整性 —— `FOREIGN KEY` (外键)

用户定义完整性 —— `CHECK`, `NOT NULL`, `UNIQUE`, `DEFAULT`

`NOT NULL` (非空)

`UNIQUE` (唯一)

`CHECK` (条件检查)

`DEFAULT` (默认值)

```sql
status INT DEFAULT 1,  -- 默认状态为 1
create_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
```

### 2. 可延迟约束（DEFERRABLE）

允许在事务提交时才检查约束，解决循环依赖问题。

```sql
-- 创建可延迟的外键约束
ALTER TABLE student 
ALTER CONSTRAINT student_dno_fkey 
DEFERRABLE INITIALLY DEFERRED;
BEGIN TRANSACTION;
DELETE FROM department WHERE dNo='03';  -- 先删学院
UPDATE student SET dNo='02' WHERE sNo='008';  -- 后改学生
COMMIT;  -- 提交时检查约束
```

 `DEFERRABLE` 两种初始状态对比

| 设置                    | 含义        | 检查时机                                                 |
| --------------------- | --------- | ---------------------------------------------------- |
| `INITIALLY IMMEDIATE` | 初始为**立即** | 每条语句执行完立刻检查（需手动 `SET CONSTRAINTS ALL DEFERRED` 才能延迟） |
| `INITIALLY DEFERRED`  | 初始为**延迟** | 事务提交时才检查                                             |

`ALTER CONSTRAINT` 在做什么？
------------------------

`ALTER CONSTRAINT` 只能修改约束的**两个布尔属性**：

| 属性           | 含义                      |
| ------------ | ----------------------- |
| `DEFERRABLE` | 约束是否可以延迟检查（创建时就得定死，不能改） |
| `INITIALLY`  | 初始是立即检查还是延迟检查           |

**注意**：`ALTER CONSTRAINT` 不能把 `NOT DEFERRABLE` 改成 `DEFERRABLE`。  必须是**创建时已经是 `DEFERRABLE`**，你才能用 `ALTER CONSTRAINT` 调整它的初始延迟状态。
CONSTRAINT 后面可以跟哪些约束？

---------------------

| 约束类型 | 写法示例                                                                 |
| ---- | -------------------------------------------------------------------- |
| 主键   | `CONSTRAINT pk_student PRIMARY KEY (sNo)`                            |
| 外键   | `CONSTRAINT fk_sc_student FOREIGN KEY (sNo) REFERENCES student(sNo)` |
| 唯一   | `CONSTRAINT uk_email UNIQUE (email)`                                 |
| 检查   | `CONSTRAINT ck_age CHECK (age >= 0)`                                 |
| 非空   | （非空通常不用 CONSTRAINT 命名，直接 `NOT NULL`）                                 |

```sql
CREATE TABLE student (
    sNo VARCHAR(10),
    dNo VARCHAR(2),
    CONSTRAINT pk_student PRIMARY KEY (sNo),
    CONSTRAINT fk_student_dno FOREIGN KEY (dNo) REFERENCES department(dNo)
);
```

## 三种本质状态（由创建时的语法决定）

| 状态                                      | 你的类比       | 能否改变检查时机？         | 默认检查时机 |
| --------------------------------------- | ---------- | ----------------- | ------ |
| **`NOT DEFERRABLE`**                    | 规则写死，立刻查   | **❌ 不能** (永久立即检查) | 每条语句后  |
| **`DEFERRABLE INITIALLY IMMEDIATE`**    | 规则可调，默认立刻查 | **✅ 能**           | 每条语句后  |
| **`DEFERRABLE INITIALLY DEFERRED`**     | 规则可调，默认下班查 | **✅ 能**           | 事务提交时  |
| **`ALTER CONSTRAINT ... DEFERRED`**     | 修改属性       |                   |        |
| **`SET CONSTRAINTS ALL/name DEFERRED`** | 修改属性       |                   |        |

### 3. 违反约束的三种操作

| 操作     | 可能违反的约束          |
| ------ | ---------------- |
| INSERT | 实体完整性、参照完整性、用户定义 |
| UPDATE | 参照完整性、用户定义       |
| DELETE | 参照完整性（被引用行被删）    |

第3-3讲：存储过程和触发器
===================

一、存储过程（Stored Procedure）
------------------------

### 1. 定义

> 一组为了完成特定功能的SQL语句集，**存储在数据库中**，一次编译后永久有效。

### 2. 特点

| 特点    | 说明                 |
| ----- | ------------------ |
| 预编译   | 首次执行后编译，后续直接调用，效率高 |
| 可重用   | 多次调用，减少网络传输        |
| 带参数   | 可传入参数，返回结果         |
| 存储在DB | 集中管理，安全性好          |

### 3. 基本语法（以PostgreSQL为例）

```sql
CREATE OR REPLACE FUNCTION 函数名(参数列表)
RETURNS 返回值类型 AS $$
BEGIN
    -- SQL语句
END;
$$ LANGUAGE plpgsql;
-- 调用
SELECT 函数名(参数);
```

```sql
-- 创建函数：根据学号查询学生姓名
CREATE OR REPLACE FUNCTION get_student_name(p_sno VARCHAR)
RETURNS VARCHAR AS $$
DECLARE
    v_name VARCHAR;
BEGIN
    SELECT sName INTO v_name
    FROM Student
    WHERE sNo = p_sno;

    RETURN v_name;
END;
$$ LANGUAGE plpgsql;
```

三种常见返回值类型
---------

```sql
-- 返回一个数字
RETURNS NUMERIC AS $$
DECLARE
    avg_score NUMERIC;

-- 返回整个查询结果
RETURNS TABLE(sNo VARCHAR, sName VARCHAR, age INT) AS $$

-- 返回一个 SETOF（多行单列）
RETURNS SETOF VARCHAR AS $$
```

### 4. 存储过程 vs 普通SQL

| 对比项 | 存储过程      | 普通SQL    |
| --- | --------- | -------- |
| 编译  | 一次编译，多次执行 | 每次执行都编译  |
| 网络  | 一次调用，传输少  | 多条语句，传输多 |
| 复用  | 可复用       | 不可复用     |
| 维护  | 集中管理      | 分散在应用程序  |

二、触发器（Trigger）
--------------

### 1. 定义

> 当对数据库执行**指定操作**时**自动触发**执行的存储过程。

### 2. 触发器的三个组成部分（必背）

| 组成部分   | 说明       | 示例                   |
| ------ | -------- | -------------------- |
| **事件** | 何时触发     | INSERT、UPDATE、DELETE |
| **条件** | 是否执行（可选） | WHEN (条件)            |
| **动作** | 触发后做什么   | 执行一段SQL/存储过程         |

### 3. 触发器类型

| 分类        | 说明          |
| --------- | ----------- |
| 行级触发器     | 每影响一行触发一次   |
| 语句级触发器    | 整个SQL语句触发一次 |
| BEFORE触发器 | 事件执行前触发     |
| AFTER触发器  | 事件执行后触发     |

### 4. 基本语法（以PostgreSQL为例）

```sql
CREATE TRIGGER 触发器名
{BEFORE | AFTER} {INSERT | UPDATE | DELETE} ON 表名
[FOR EACH {ROW | STATEMENT}]  --行级触发 or 语句级触发
[WHEN (条件)]
EXECUTE FUNCTION 函数名();
-- 示例
CREATE TRIGGER update_log_trigger
AFTER UPDATE ON Student
FOR EACH ROW
EXECUTE FUNCTION log_student_change();
```

### 5. 触发器的作用

| 作用    | 说明              |
| ----- | --------------- |
| 审计日志  | 自动记录数据变更        |
| 数据完整性 | 复杂约束无法用CHECK实现时 |
| 自动计算  | 如更新总分、计数        |
| 同步复制  | 多表同步            |

### 6. 触发器 vs 存储过程

| 对比项  | 触发器       | 存储过程      |
| ---- | --------- | --------- |
| 调用方式 | **自动触发**  | 显式调用      |
| 参数   | 不支持传入参数   | 支持        |
| 返回值  | 无直接返回值    | 可返回值      |
| 适用场景 | 数据变更时自动处理 | 主动执行的业务逻辑 |

第3-5讲：查询性能分析与优化 完整总结
====================

一、SQL查询处理过程
-----------

* **查询块**：SQL语句分解为的基本单元，可转换为关系代数表达式

* **查询树**：关系代数表达式对应的树状结构（叶节点=输入关系，中间节点=运算）

* **基于启发式规则（ Heuristic Rule ）**：在查询执行策略中对运算（基于规则）进行排序和调整。

* **基于系统估算（Systematically Estimating）或开销**：基于不同执行策略的代价估算，选择代价最小的执行策略。

二、基本操作的实现（重点）
-------------

### 1. 选择运算的实现方法

| 方法             | 说明        | 适用场景      |
| -------------- | --------- | --------- |
| **线性搜索（全表扫描）** | 逐行检查      | 无条件或小表    |
| **二分查找**       | 要求排序      | 等值条件，有序   |
| **索引**         | B+树、Hash等 | 有索引，高选择性  |
| **B+树索引**      | 多层索引      | 范围查询、等值查询 |

### 2. 连接运算的实现方法（高频考点）

| 方法         | 原理            | 复杂度                  | 适用场景       |
| ---------- | ------------- | -------------------- | ---------- |
| **嵌套循环连接** | 外层每个元组遍历内层整个表 | O(n×m)               | 小表×小表      |
| **排序合并连接** | 两边排序后合并       | O(n log n + m log m) | 大表，已排序或可排序 |
| **索引连接**   | 利用内表索引查找      | O(n × log m)         | 内表有索引      |
| **Hash连接** | 建Hash表后匹配     | O(n+m)               | 等值连接，大表    |

### 3. 连接方法对比表

| 方法     | 是否需要排序 | 是否需要索引 | 最佳场景        |
| ------ | ------ | ------ | ----------- |
| 嵌套循环   | 否      | 否      | 小表驱动大表      |
| 排序合并   | 是      | 否      | 两个大表等值连接    |
| 索引连接   | 否      | 是      | 内表有索引       |
| Hash连接 | 否      | 否      | 大表等值连接，内存足够 |

三、启发式优化规则
------------

| 规则                  | 说明        |
| ------------------- | --------- |
| **1. 选择尽可能先做**      | 减少后续操作数据量 |
| **2. 投影与选择同时进行**    | 减少处理列数    |
| **3. 投影与其前后双目运算结合** | 减少中间结果    |
| **4. 选择与笛卡尔积结合为连接** | 避免显式笛卡尔积  |
| **5. 找出公共子表达式**     | 避免重复计算    |

五、优化器代价估算
---------

### 需要统计的信息（数据字典中存储）

| 信息类型  | 内容             |
| ----- | -------------- |
| 基本表信息 | 元组数N、块数B、元组长度  |
| 列信息   | 不同值个数m、最大值、最小值 |
| 索引信息  | 层数L、叶子节点数、选择基数 |

### 总代价公式

总代价 = I/O代价 + CPU代价 + 内存代价 + 通信代价

> 通常 **I/O代价** 占主导（磁盘读写最慢）

第4-1讲：数据库设计概述
=============

一、数据库系统生存周期
-----------

### 1. 生存周期各阶段（按顺序）

| 阶段          | 核心任务               |
| ----------- | ------------------ |
| **数据库规划**   | 明确任务陈述和任务目标，制定标准   |
| **系统定义**    | 描述应用范围、边界和主要用户视图   |
| **需求收集与分析** | 收集用户需求，输出需求规格说明书   |
| **数据库设计**   | 概念设计 → 逻辑设计 → 物理设计 |
| **DBMS选择**  | 选择适合的数据库管理系统       |
| **应用程序设计**  | 设计用户界面和事务          |
| **原型构建**    | 快速构建原型，获取用户反馈      |
| **实现**      | 编写代码，创建数据库         |
| **数据转换与加载** | 将旧系统数据迁移到新数据库      |
| **测试**      | 功能测试、性能测试          |
| **运行维护**    | 日常运维、性能调优、需求变更     |

> 记忆提示：**规系需设选，应原实加测运**

二、用户视图（User View）
-----------------

* **定义**：从特定用户角度看到的数据库应用需求（需要哪些数据，执行哪些操作）

* 一个数据库应用可以有**一个或多个用户视图**

* 不同用户视图之间可能**重叠**

三、数据库设计的6个阶段
------------

### 阶段1：需求分析

| **方法** | 结构化分析（SA），自顶向下逐层分解 |
| ------ | ------------------ |
| **输出** | 数据字典、数据流图、需求规格说明书  |

### 阶段2：概念结构设计

| **任务** | 将用户需求抽象为**独立于任何DBMS**的概念模型 |
| ------ | -------------------------- |
| **输出** | E-R图（实体-联系图）               |

### 阶段3：逻辑结构设计

| **任务** | 将概念模型（E-R图）转换为特定数据模型（如**关系模型**） |
| ------ | ------------------------------- |
| **输出** | 关系模式（表）、视图（外模式）                 |
| **特点** | 与具体DBMS无关，但与数据模型类型相关            |

### 阶段4：物理结构设计

| **任务** | 确定数据库在存储介质上的**存储结构和存取方法**   |
| ------ | --------------------------- |
| **输出** | 内模式（索引、分区、聚簇等）              |
| **特点** | **与具体DBMS相关**，需要考虑存储、索引、I/O |

### 阶段5：数据库实施

| **任务** | 用DDL创建数据库结构，编写应用程序，组织数据入库 |
| ------ | ------------------------- |
| **输出** | 运行中的数据库和应用                |

第4-3讲：关系模式规范化理论-1
==================
一、为什么需要规范化

----------

### 1. 不规范化带来的问题

| 问题       | 说明                 |
| -------- | ------------------ |
| **数据冗余** | 同一数据重复存储，浪费空间      |
| **更新异常** | 修改一处数据需要多处修改，容易不一致 |
| **插入异常** | 某些数据无法插入（因为其他数据为空） |
| **删除异常** | 删除数据时会丢失其他相关信息     |

### 2. 规范化的目的

* 通过**模式分解**，将低范式关系模式转换为高范式关系模式的集合

* 消除上述异常

### 3. 范式层次（从低到高）

1NF → 2NF → 3NF → BCNF → 4NF → 5NF

| 范式   | 主要解决的问题                |
| ---- | ---------------------- |
| 1NF  | 属性原子性                  |
| 2NF  | 消除非主属性对主码的**部分依赖**     |
| 3NF  | 消除非主属性对主码的**传递依赖**     |
| BCNF | 消除所有属性对候选码的传递依赖（包括主属性） |

### 3. 函数依赖与联系类型的对应关系

| 联系类型 | 函数依赖关系             |
| ---- | ------------------ |
| 1:1  | X ↔ Y（X→Y 且 Y→X）   |
| 1:N  | Y → X **（N方决定1方）** |
| M:N  | 无函数依赖              |

### 4. 函数依赖的分类

| 类型        | 定义                          | 示例                           |
| --------- | --------------------------- | ---------------------------- |
| **平凡FD**  | B ⊆ A                       | (学号, 姓名) → 学号                |
| **非平凡FD** | B ⊄ A 且 B ≠ ∅               | 学号 → 姓名                      |
| **完全FD**  | X → Y，且不存在 X' ⊂ X 使得 X' → Y | (学号, 课程号) → 成绩（单独学号不能决定成绩）   |
| **部分FD**  | X → Y，且存在 X' ⊂ X 使得 X' → Y  | (学号, 课程号) → 姓名（学号单独就能决定姓名）   |
| **传递FD**  | X → Y，Y → Z，且 Y → X 不成立     | 学号 → 学院，学院 → 院长，则学号 → 院长（传递） |

三、Armstrong公理（推理规则）
---------------------

### 1. 基本公理

| 规则                      | 名称  | 说明      |
| ----------------------- | --- | ------- |
| 若 B ⊆ A，则 A → B         | 自反律 | 平凡FD    |
| 若 A → B，则 AC → BC       | 增广律 | 两边加相同属性 |
| 若 A → B 且 B → C，则 A → C | 传递律 | 链式推导    |

### 2. 导出规则

| 规则                        | 名称  | 说明     |
| ------------------------- | --- | ------ |
| A → A                     | 自决定 | 自反律特例  |
| 若 A → BC，则 A → B 且 A → C  | 分解律 | 右边拆开   |
| 若 A → B 且 A → C，则 A → BC  | 合并律 | 右边合并   |
| 若 A → B 且 C → D，则 AC → BD | 组合律 | 左右分别合并 |

四、属性闭包（Closure of Attributes）
--------------------------------

### 1. 定义

> 给定函数依赖集 F，**属性集 X 的闭包 X⁺** 是由 X 通过 F 能推导出的所有属性的集合

### 2. 求闭包的算法（重点）

**步骤**：

1. 初始化 X⁽⁰⁾ = X

2. 重复：找 F 中左部 ⊆ 当前 X⁽ⁱ⁾ 的 FD，将其右部加入 X⁽ⁱ⁾

3. 直到 X⁽ⁱ⁾ 不再增大

### 3. 闭包的作用

| 用途            | 方法                    |
| ------------- | --------------------- |
| 判断 X → Y 是否成立 | 检查 Y ⊆ X⁺             |
| 求候选码          | 找最小的 X 使得 X⁺ = U（全属性） |

五、最小函数依赖集（Minimal Cover）
--------------------------

### 1. 三个条件

| 条件          | 说明           |
| ----------- | ------------ |
| **右部单属性**   | 每个FD右边只有一个属性 |
| **无冗余依赖**   | 不能由其他FD推导出来  |
| **左部无冗余属性** | 每个FD左边没有多余属性 |

### 2. 例题

```markdown
F = {
  sNo → sName,
  sNo → sDept,
  sNo → deptMn,      ← 注意：deptMn 实际应由 sDept 决定
  sDept → deptAddr,
  sDept → deptMn,
  cNo → cName,
  (sNo, cNo) → score
}
```

**最小依赖集**：

```markdown
Fmin = {
  sNo → sName,
  sNo → sDept,
  sDept → deptAddr,
  sDept → deptMn,
  cNo → cName,
  (sNo, cNo) → score
}
```

第4-4讲：E-R模型向关系模型的映射
===================

一、映射目标
------

将E-R图中的元素转换为**关系模式（表）+ 约束（主码、外码）**

| E-R模型元素 | 关系模型元素  |
| ------- | ------- |
| 实体      | 表       |
| 属性      | 列       |
| 联系      | 外键（或新表） |
| 键属性     | 主键      |
| 值集      | 域       |

二、映射前的预处理（4种需要拆解的情况）
--------------------

在正式映射前，需要先将以下4种E-R元素**拆解**，否则无法直接映射：

| 待拆解元素           | 拆解方法                    |
| --------------- | ----------------------- |
| **M:N 二元联系**    | 创建一个新表（中间表），包含双方实体的主键   |
| **M:N 递归联系**    | 创建一个新表，包含两列，都引用同一实体的主键  |
| **复杂联系（三元及以上）** | 创建一个新表，包含所有参与实体的主键      |
| **多值属性**        | 创建一个新表，包含主实体的主键 + 该多值属性 |

三、正式映射规则（按顺序）
-------------

### 1. 强实体

| 规则               | 说明              |
| ---------------- | --------------- |
| 每个强实体创建一个表       | 表名 = 实体名        |
| 包含所有**简单属性**     | 复合属性只拆入其**子属性** |
| 主键 = 实体的**主键属性** |                 |

**示例**：

```
实体：Staff(staffNo, fName, lName, position, sex, DOB)
主键：staffNo
```

* * *

### 2. 弱实体

| 规则                          | 说明        |
| --------------------------- | --------- |
| 每个弱实体创建一个表                  | 表名 = 弱实体名 |
| 主键 = **弱实体部分主键 + 所属强实体的主键** |           |
| 所属强实体的主键作为**外键**            |           |

**示例**：

* 强实体：Employee(empNo, name)

* 弱实体：Dependent(name, birthDate)

* 联系：Employee 拥有 Dependent

* 映射结果：Dependent(**empNo, name**, birthDate)  
  （主键 = empNo + name，empNo 同时是外键）

* * *

### 3. 1:N 二元联系（最重要 ⭐⭐）

| 规则                          | 说明  |
| --------------------------- | --- |
| 将**1方**的主键作为外键，加入到**N方**的表中 |     |
| 联系的属性也放入 N 方的表中             |     |

**示例**：

```
Department(dNo, dName)  ← 1方
Employee(eNo, eName)    ← N方
```

映射后：

```
Employee(eNo, eName, dNo)  ← dNo 是外键
```

### 4. 1:1 二元联系（分3种情况）

#### 情况A：双方都是**强制参与**

* **合并**为一个表

* 选择其中一个实体的主键作为新表主键，另一个作为候选键

**示例**：

```
Student(studentNo, name)
Like(likeName, level)
联系：Student 拥有 Like（1:1，双方必须）
```

合并结果：

```
Student(studentNo, name, likeName, level)
主键：studentNo
```

#### 情况B：一方强制，一方可选

* **可选方**作为父实体（外键来源）

* 将可选方的主键放入**强制方**的表中作为外键

**示例**：

```
Teacher(tNo, name)  ← 可选（教师不一定有车）
Car(carNo, color)   ← 强制（车必须有主）
```

映射结果：

```
Teacher(tNo, name, carNo)  ← carNo 是外键
Car(carNo, color)
```

#### 情况C：双方都是**可选参与**

* **任意选择**一个实体的主键放入另一个表（或新建关系表）

* 通常新建一个独立的关系表更灵活

* * *

### 5. 1:1 递归联系

| 参与情况   | 映射方法          |
| ------ | ------------- |
| 双方强制参与 | 单表，放两列主键副本    |
| 一方强制参与 | 单表（两列）或 新建关系表 |
| 双方可选参与 | 新建关系表         |

**示例（员工-经理递归联系）**：

```
Employee(empNo, name)
联系：manage（员工管理员工）
```

映射（新建关系表）：

```
Manage(managerNo, employeeNo)
外键：managerNo, employeeNo 都引用 Employee(empNo)
```

* * *

### 6. M:N 二元联系（已预处理）

创建**新表**，包含：

* 双方实体的主键（作为外键）

* 联系的属性

* 主键 = 双方主键的组合

**示例**：

```
Student(sNo, name)
Course(cNo, cName)
联系：Select（M:N，有score属性）
```

映射结果：

```
Select(sNo, cNo, score)
主键：(sNo, cNo)
外键：sNo → Student(sNo)，cNo → Course(cNo)
```

### 7. 超类/子类联系（EER）

| 约束类型                | 映射方法                |
| ------------------- | ------------------- |
| **M + And**（强制+不相交） | 单一表：包含所有属性 + 判别属性   |
| **O + Or**（可选+重叠）   | 两张表：超类表 + 子类表（共享主键） |
| **M + Or**（强制+重叠）   | 多张表：每种子类一张表（含超类属性）  |
| **O + And**（可选+不相交） | 两张表：超类表 + 子类表       |

> 实际考试中常简化为：**通常创建超类表 + 子类表，子类表的主键也是外键引用超类表**

### 情况一：`M + And`（强制 + 不相交）

> 每个人**必须**是学生或教师，且不能同时是两者。

**映射**：**一张表**

### 情况二：`O + Or`（可选 + 重叠）

> 一个人**可以**只是普通人员，也可以是学生，也可以是教师，也可以两者都是。

**映射**：**两张表**(超类：Person+子类：teacher+student)

* 一个人如果在 Student 表里有记录，说明他是学生

* 可以同时在 Student 和 Teacher 表（重叠）

* 可以都不在（只是普通人员）

### 情况三：`M + Or`（ 强制重叠）

> 每个人**必须**是学生或教师，且可以同时是两者。

**映射**：**多张表，每种子类一张表（含超类属性）**

-- 没有单独的 Person 表
-- 学生表包含所有人员属性

-- 教师表也包含所有人员属性

### 情况四：`O + And`（可选不相交）

> 一个人**可以**是学生或教师，但**不能**同时是两者。

**映射**：**两张表**（同上情况二的写法）

### 8. 多值属性（已预处理）

创建**新表**，包含：

* 主实体的主键（作为外键）

* 多值属性本身

* 主键 = 主实体主键 + 多值属性

**示例**：

```
Employee(empNo, name)
多值属性：phoneNo
```

映射结果：

```
Employee_Phone(empNo, phoneNo)
主键：(empNo, phoneNo)
外键：empNo → Employee(empNo)
```

第4-4讲：关系模式规范化理论-2
=================

一、第二范式（2NF）
----------------

### 1. 2NF定义

> 满足1NF，且**所有非主属性完全函数依赖于主码**（无部分依赖）

### 3. 2NF分解方法

将部分依赖的属性连同其决定因素一起拆出：

**原模式**：SC(sNo, cNo, score, credit)  
**分解后**：

* SC(sNo, cNo, score)

* Course(cNo, credit)

### 4. 重要结论

> **单属性主码的关系模式一定满足2NF**

二、第三范式（3NF）
--------------

### 1. 传递依赖的定义

> 若 A → B，B → C，且 B → A 不成立（即B不是决定A的键），则 C 传递依赖于 A

**形式化**：A → B，B → C，且 B ⊅ A，则 A → C 是传递依赖

### 2. 3NF定义

> 满足2NF，且**没有非主属性传递依赖于任何候选码**

### 3. 3NF分解方法

将传递依赖的属性连同其决定因素一起拆出：

**原模式**：SD(sNo, sName, dept, deptAddr, deptMn)  

**分解后**：

* S(sNo, sName, dept)

* Dept(dept, deptAddr, deptMn)

三、2NF与3NF的关系
------------------

### 1. 包含关系

> **3NF ⊂ 2NF**（满足3NF一定满足2NF，反之不成立）

四、BCNF（Boyce-Codd Normal Form）
---------------------------------

### 1. 为什么需要BCNF

3NF仍然可能存在问题：**主属性之间的传递依赖**

### 2. BCNF定义

> 对于关系模式R中**每一个非平凡函数依赖 X → Y**，X 都**必须包含候选码**（即X是超码）

换句话说：**所有决定因素都必须是候选码**

**经典案例**：STJ(S, T, J) —— 学生、教师、课程

* 语义：每名教师只教一门课；每门课有若干教师；学生选定某门课对应固定教师

* 函数依赖：(S,J)→T, (S,T)→J, T→J

* 候选码：(S,J) 和 (S,T)

* **问题**：T→J 中，T 不是候选码（T单独不能决定S）

* **结论**：STJ ∈ 3NF，但 **STJ ∉ BCNF**

五、范式递进关系总结
----------

```
1NF（属性原子）
   ↓ 消除部分依赖
2NF（非主属性完全依赖主码）
   ↓ 消除传递依赖
3NF（非主属性不传递依赖主码）
   ↓ 消除主属性对非主码的决定因素
BCNF（所有决定因素都是候选码）
   ↓ 消除多值依赖
4NF（无多值依赖）
```

六、第四范式（4NF）
-----------

### 1. 多值依赖（MVD）

> A →→ B：对于A的每个值，有一组B值与之对应，且与C无关

> **多值依赖 `A →→ B` 的本质是：A 决定 B 的一个“集合”，并且 B 和 C 完全独立，导致表格里必须把 B 和 C 做“排列组合”来存数据。解决方法是拆分表格。**

第5-1讲：事务模型
==========

 一、事务的概念

### 1. 事务的定义

> **事务**是数据库操作的**逻辑工作单元**，由单个用户或应用程序执行的一系列操作（读/写数据库）。

### 2. 事务的执行环境与风险

| 风险类型        | 说明                   |
| ----------- | -------------------- |
| **单事务风险**   | 事务执行过程中可能发生系统故障、介质故障 |
| **多事务并发风险** | 多个事务同时执行可能导致数据不一致    |

> DBMS通过**事务模型**隐藏并发和故障的复杂性，程序员只需关注业务逻辑。

二、事务的ACID特性（必背 ⭐⭐⭐）
-------------------

| 特性      | 含义               | 实现机制      |
| ------- | ---------------- | --------- |
| **原子性** | 事务中的操作要么全做，要么全不做 | 恢复协议      |
| **一致性** | 事务执行前后数据库保持完整性约束 | 完整性约束、触发器 |
| **隔离性** | 并发事务之间互不干扰       | 并发控制协议    |
| **持久性** | 已提交事务的修改永久保存     | 恢复协议      |

第5-2讲：并发控制-1
============

### 1. 丢失更新（Lost Update）

| 时间  | T1                | T2                  | bal_x |
| --- | ----------------- | ------------------- | ----- |
| t1  | begin             |                     | 100   |
| t2  | read(bal_x)=100   |                     | 100   |
| t3  | bal_x=bal_x-10=90 | begin               | 100   |
| t4  | write(bal_x)=90   | read(bal_x)=100     | 100   |
| t5  | commit            | bal_x=bal_x+100=200 | 90    |
| t6  |                   | write(bal_x)=200    | 200   |
| t7  |                   | commit              | 200   |

**问题**：T1的更新被T2覆盖，丢失了T1的修改（90 → 200，T1的-10丢失）

### 2. 脏读（Dirty Read）

| 时间  | T1                  | T2                 | bal_x |
| --- | ------------------- | ------------------ | ----- |
| t1  | begin               |                    | 100   |
| t2  | read(bal_x)=100     |                    | 100   |
| t3  | bal_x=bal_x+100=200 |                    | 100   |
| t4  | write(bal_x)=200    | begin              | 200   |
| t5  |                     | read(bal_x)=200    | 200   |
| t6  | rollback（恢复为100）    | bal_x=bal_x-10=190 | 100   |
| t7  |                     | write(bal_x)=190   | 190   |
| t8  |                     | commit             | 190   |

**问题**：T2读到了T1**未提交**的数据，T1回滚后T2读到的数据是无效的。

### 3. 不可重复读/幻读（Non-Repeatable Read）

| 时间  | T3                | T4                    | bal_x |
| --- | ----------------- | --------------------- | ----- |
| t1  | begin             |                       | 10000 |
| t2  | read(bal_x)=10000 | begin                 | 10000 |
| t3  |                   | read(bal_x)=10000     | 10000 |
| t4  |                   | bal_x=bal_x+100=10100 | 10000 |
| t5  |                   | write(bal_x)=10100    | 10100 |
| t6  |                   | commit                | 10100 |
| t7  | read(bal_x)=10100 |                       | 10100 |

**问题**：T3中两次读取bal_x，得到不同结果（10000 → 10100），因为T4修改后提交了。

### 4. 不一致分析（幻读诱发）

| 时间  | T5                 | T6                | 说明  |
| --- | ------------------ | ----------------- | --- |
| t1  | begin              | begin             |     |
| t2  | sum=0              | read(bal_x)=100   |     |
| t3  | read(bal_x)=100    | bal_x=bal_x-10=90 |     |
| t4  | sum=100            | write(bal_x)=90   |     |
| t5  | read(bal_y)=50     |                   |     |
| t6  | sum=150            |                   |     |
| t7  | read(bal_z)=25     | bal_z=bal_z+10=35 |     |
| t8  | sum=175            | write(bal_z)=35   |     |
| t9  |                    | commit            |     |

**问题**：T6在T5执行过程中插入/修改了数据，导致T5的统计结果包含了T6的修改，出现幻象。

五、冲突可串行化（Conflict Serializable）

----------------------------------

### 1. 冲突操作

| 操作1 | 操作2 | 是否冲突  |
| --- | --- | ----- |
| 读A  | 读A  | ❌ 不冲突 |
| 读A  | 写A  | ✅ 冲突  |
| 写A  | 读A  | ✅ 冲突  |
| 写A  | 写A  | ✅ 冲突  |

**结论**：只有**同一数据**的**读-写、写-读、写-写**才冲突；**读-读**不冲突。

**什么叫“冲突操作”？**  

两个操作**同时满足以下三个条件**才算冲突：

1. 属于**不同事务**

2. 操作**同一个数据项**

3. 至少有一个是**写操作**（`write`）

**冲突操作不能随便交换顺序**，交换后结果可能不一样。

**非冲突操作**可以随便交换，不影响结果。

### 2. 冲突可串行化定义

> 若调度S可以通过**交换非冲突操作**转换为某个串行调度，则S是**冲突可串行化**的。

### 3. 判断方法（优先图/拓扑排序）

* 节点 = 事务

* 边 Ti → Tj 表示 Ti 的操作在冲突中先于 Tj

* 若无环 → 冲突可串行化

### 案例 1：标准无环例子

调度 S：

text

T1: read(A), write(A)
T2: read(A), write(A)

顺序：`T1 read(A)` → `T1 write(A)` → `T2 read(A)` → `T2 write(A)`

找冲突：

| 冲突                        | 谁先  | 边       |
| ------------------------- | --- | ------- |
| T1 write(A) 与 T2 read(A)  | T1  | T1 → T2 |
| T1 write(A) 与 T2 write(A) | T1  | T1 → T2 |

**冲突图**：只有 T1 → T2，**无环** ✅

**结论**：冲突可串行化（等价于串行顺序 T1 → T2）。

* * *

### 案例 2：有环（不可串行化）

调度 S：

text

T1: read(Q), write(Q)
T2: write(Q)

顺序：`T1 read(Q)` → `T2 write(Q)` → `T1 write(Q)`

冲突：

| 冲突                        | 谁先  | 边       |
| ------------------------- | --- | ------- |
| T1 read(Q) 与 T2 write(Q)  | T1  | T1 → T2 |
| T2 write(Q) 与 T1 write(Q) | T2  | T2 → T1 |

**冲突图**：T1 → T2 且 T2 → T1 → **有环** ❌

**结论**：不是冲突可串行化。
第5-2讲：并发控制-2
=============
一、冲突可串行化的判定——优先图（Precedence Graph）

-------------------------------------

### 1. 优先图定义

* **节点**：每个事务是一个顶点

* **边**：从 Ti 到 Tj 的有向边，表示 Ti 在冲突中先于 Tj

### 2. 添加边的条件（冲突的三种情况）

| 条件            | 说明          | 边方向     |
| ------------- | ----------- | ------- |
| Ti 写 Q，Tj 读 Q | Ti 先写，Tj 后读 | Ti → Tj |
| Ti 读 Q，Tj 写 Q | Ti 先读，Tj 后写 | Ti → Tj |
| Ti 写 Q，Tj 写 Q | Ti 先写，Tj 后写 | Ti → Tj |

> **注意**：读-读不冲突，不加边

### 3. 判定规则

> **优先图无环 ↔ 调度是冲突可串行化的**
> 
> **时间复杂度**：O(N²)

二、两段锁协议（2PL）
------------

### 1. 为什么需要2PL

* 实际系统中，操作系统决定事务操作的交叉顺序，DBMS无法控制

* 因此使用**协议**来保证可串行化，而不是事后检测

### 2. 锁的类型

| 锁类型     | 符号        | 功能      | 兼容性        |
| ------- | --------- | ------- | ---------- |
| **共享锁** | LOCK_S(Q) | 只读，不能修改 | 可与另一个共享锁兼容 |
| **专用锁** | LOCK_X(Q) | 可读可写    | 不与其他任何锁兼容  |

**锁兼容矩阵**：

| 请求锁 | 已持有共享锁 | 已持有专用锁 |
| --- | ------ | ------ |
| 共享锁 | ✅ 兼容   | ❌ 不兼容  |
| 专用锁 | ❌ 不兼容  | ❌ 不兼容  |

![d1a14d07-c3ac-480e-b1a7-fa4f4d5a9a81](file:///C:/Users/15829/Pictures/Typedown/d1a14d07-c3ac-480e-b1a7-fa4f4d5a9a81.png)

### 3. 两段锁协议定义

> 每个事务中，**所有加锁操作必须在第一个解锁操作之前**

PS：加锁和解锁其实没有语法上硬性的要求，只不过两端锁协议要求所有加锁操作都在第一个解锁操作之前

**两个阶段**：

| 阶段       | 行为          |
| -------- | ----------- |
| **增长阶段** | 只能加锁，不能释放锁  |
| **收缩阶段** | 只能释放锁，不能加新锁 |

![3919476e-9e33-441d-985c-280d6c3f04c7](file:///C:/Users/15829/Pictures/Typedown/3919476e-9e33-441d-985c-280d6c3f04c7.png)

### 4. 2PL的充分性

> 如果调度中每个事务都遵循2PL，则调度**一定是冲突可串行化的**

**注意**：这是**充分条件**，不是必要条件（不遵循2PL也可能可串行化）

### 5. 2PL存在的问题

| 问题          | 说明                   |
| ----------- | -------------------- |
| **级联回滚**    | T1回滚可能导致依赖T1的T2也必须回滚 |
| **难以预测收缩点** | 事务不知道何时不再需要锁         |

### 1. 为什么会有“级联回滚”？（脏读问题）

**核心原因：** 在基础2PL中，事务在**收缩阶段**（提交前）就会释放锁。如果此时事务还没提交，其他事务读到了它修改的数据，一旦前者回滚，后者就白读了（读到了脏数据），后者也得被迫回滚。

**举个例子（你 PDF 第 13 页的图）：**

* **T1**：A 账户 +100 元（修改数据，没提交）。

* **T2**：读取 A 账户（因为 T1 释放了锁，T2 读到新余额）。

* **T1 回滚**：A 账户恢复到原状。

* **结果**：T2 读到的 200 元其实是“幻觉”，它基于这个错误数据做了操作，**T2 也必须跟着回滚**。如果还有 T3 依赖 T2，就会像多米诺骨牌一样全部回滚，这就是**级联回滚**。

### 2. 为什么“难以预测收缩点”？（工程实现难题）

**核心原因：** 2PL 要求事务在进入“收缩阶段”（开始释放锁）**之前**，必须获得**所有**需要的锁。但问题是：事务在运行前，通常**不知道自己到底需要哪些锁**

* **情况 A（程序逻辑）**：`if (条件) { 读表B } else { 读表C }`。执行前，不知道会走哪条路，也就不知道要不要锁 B 或 C。

* **情况 B（用户交互）**：一个长事务，先读数据展示给用户，用户思考 5 分钟后再决定改哪条。这 5 分钟里，锁还占着不放，效率极低。

如果让程序员预估所有可能用到的锁，只能**全部锁一遍**（锁住整个数据库），性能极差。这就是“难以预测收缩点”。

### 解决方案

这两个问题促使数据库引入了**严格两段锁协议（Rigorous 2PL）**

| 问题        | 基础 2PL 的表现     | **严格 2PL (Rigorous 2PL)** 的解决方案 |
| --------- | -------------- | ------------------------------- |
| **级联回滚**  | 提交前释放锁，导致脏读    | **提交后才释放所有锁**（杜绝脏读，完全避免级联回滚）    |
| **预测收缩点** | 需提前预测所有锁，否则性能差 | 不用预测了。**锁直接保留到提交**，需要时就加锁，不需要预测 |

三、严格两段锁协议（Rigorous 2PL）
-------------------------

### 1. 定义

> 事务在操作数据项**之前才申请锁**，并且**所有锁（特别是专用锁）一直保持到事务提交/回滚才释放**

### 2. 与普通2PL的区别

| 对比项    | 普通2PL      | 严格2PL      |
| ------ | ---------- | ---------- |
| 解锁时机   | 增长阶段后随时可解锁 | 只在提交/回滚时解锁 |
| 避免级联回滚 | ❌          | ✅          |
| 可恢复性   | 不一定保证      | 保证         |

### 3. 严格2PL的作用

* 保证调度是**可恢复的**（Recoverable）

* 避免脏读和级联回滚

> 大多数DBMS实现的是**严格2PL**或**强严格2PL**

四、死锁（Deadlock）

----------------

### 1. 死锁定义

> 两个或多个事务**相互等待对方持有的锁**，导致无法继续执行

### 2. 死锁示例

| 时间  | Ta             | Tb             |
| --- | -------------- | -------------- |
| t1  | Lock_X(A) ✅    |                |
| t2  | read(A)        | Lock_X(B) ✅    |
| t3  | A=A-10         | read(B)        |
| t4  | write(A)       | B=B+100        |
| t5  | Lock_X(B) ❌ 等待 | write(B)       |
| t6  | 等待             | Lock_X(A) ❌ 等待 |

**结果**：Ta等Tb释放B，Tb等Ta释放A → **死锁**

### 3. 死锁的解决方法

| 方法          | 说明                  |
| ----------- | ------------------- |
| **死锁预防**    | 事务在开始前一次性申请所有锁（代价大） |
| **死锁检测与解除** | 周期检测，发现死锁后回滚其中一个事务  |

### 4. 等待图（Wait-for Graph）

* **节点**：事务

* **边**：Ti → Tj 表示 Ti 在等待 Tj 释放锁

* **死锁条件**：等待图中有环

**时间复杂度**：O(N²)

一、事务边界设计

--------

### 1. 什么是事务边界

> 决定事务**从哪里开始**、**到哪里结束**的设计决策

### 2. 两种设计方式对比

| 对比项   | 方式一（长事务） | 方式二（短事务） |
| ----- | -------- | -------- |
| 事务开始  | 用户登录时    | 执行操作前    |

| 适用场景  | 简单系统     | 高并发系统    |

二、程序变量一致性
---------

### 问题场景

* 程序中的变量（如`amount`）可能在事务执行过程中被修改

* 事务回滚时，程序变量不会自动回滚

### 解决方案

* 事务回滚时，**程序变量也要重置**

* 或者只在事务**提交成功后**才更新程序变量

第5-4讲：数据库恢复
===========

三、数据库恢复概述
---------

### 1. DBMS执行环境及潜在故障

| 故障类型 | 说明             |
| ---- | -------------- |
| 事务故障 | 事务内部错误、死锁、主动回滚 |
| 系统故障 | 操作系统崩溃、断电      |
| 介质故障 | 磁盘损坏、数据丢失      |

### 2. 恢复的基本操作

| 操作       | 作用         | 对应ACID |
| -------- | ---------- | ------ |
| **Undo** | 撤销未提交事务的修改 | 原子性    |
| **Redo** | 重做已提交事务的修改 | 持久性    |

四、更新策略与传播策略
-----------

### 1. 更新策略（数据何时写入数据库）

| 策略            | 说明           | 对应操作    |
| ------------- | ------------ | ------- |
| **Immediate** | 操作执行后立即写入数据库 | 需要Undo  |
| **Deferred**  | 事务提交时才写入数据库  | 不需要Undo |

### 2. 传播策略（提交时是否强制写盘）

| 策略           | 说明           | 对应操作    |
| ------------ | ------------ | ------- |
| **Force**    | 提交时强制将修改写入磁盘 | 不需要Redo |
| **No-Force** | 提交时不一定写入磁盘   | 需要Redo  |

### 3. 四种组合

| 更新策略      | 传播策略     | Undo需要 | Redo需要 |
| --------- | -------- | ------ | ------ |
| Immediate | Force    | ✅      | ❌      |
| Immediate | No-Force | ✅      | ✅      |
| Deferred  | Force    | ❌      | ❌      |
| Deferred  | No-Force | ❌      | ✅      |

> 实际系统最常用：**Immediate + No-Force**（需要Undo和Redo）

五、日志（Log）
------------

### 1. 日志的作用

> 记录事务执行的**历史信息**，用于Undo和Redo

### 2. 日志记录格式（常见）

```
[Ti, 开始]
[Ti, D, 旧值, 新值]   ← 更新记录
...
[Ti, 提交] 或 [Ti, 回滚]
```

### 3. 两条核心原则（WAL原则 + Redo原则）

| 原则            | 名称        | 规则                   | 目的    |
| ------------- | --------- | -------------------- | ----- |
| **Undo Rule** | WAL（预写日志） | 数据库修改必须在对应日志落盘**之后** | 保证原子性 |
| **Redo Rule** | —         | 事务提交必须在事务日志落盘**之后**  | 保证持久性 |

> **WAL（Write-Ahead Logging）**：先写日志，后写数据库

### 步骤1：分析阶段

* 从日志末尾**反向扫描**

* 确定：哪些事务已提交、哪些已回滚、哪些在故障时仍活跃

### 步骤2：Undo阶段

* 回滚所有**未提交**事务的修改

* 按**反向顺序**执行Undo

### 步骤3：Redo阶段

* 重做所有**已提交**事务的修改

* 按**正向顺序**执行Redo

> 注意：Undo和Redo需要是**幂等**的（多次执行结果相同）

七、检查点（Checkpoint）
-------------------

### 1. 为什么需要检查点

* 没有检查点时，恢复需要扫描**整个日志**，效率极低

* 检查点可以**截断日志**，只保留检查点之后的内容

### 2. 检查点的工作方式

1. 定期将**内存中的修改**刷入磁盘

2. 在日志中记录检查点标记

3. 检查点之前已提交的事务不需要Redo

4. 恢复时从最近的检查点开始处理

### 3. 检查点的作用

| 作用     | 说明             |
| ------ | -------------- |
| 减少恢复时间 | 只需要处理检查点之后的事务  |
| 缩小日志   | 可以安全删除检查点之前的日志 |

* * *

八、介质故障恢复
--------

### 方法

| 方法       | 说明       |
| -------- | -------- |
| **静态备份** | 数据库关闭时备份 |
| **动态备份** | 数据库运行时备份 |
| **归档日志** | 保存所有历史日志 |
| **重做**   | 用备份+日志恢复 |

> 对于介质故障，通常需要：**最近的全量备份 + 所有后续日志**

第6-1讲：分布式数据库系统
=================

一、分布式数据库的基本概念
-------------

### 1. 定义

> **分布式数据库系统（DDBS）**：地理上分散、逻辑上统一的数据库系统。通过计算机网络将地理上分散的各局域节点连接起来，共同组成一个逻辑上统一的数据库系统。

**核心特点**：计算机网络技术 + 数据库技术的结合

### 2. 组成要素

| 术语                    | 定义                      |
| --------------------- | ----------------------- |
| **节点/场地（Node/Site）**  | 网络中的逻辑单位，可以是单台计算机或局域网   |
| **分布式数据库（DDB）**       | 分布在计算机网络上的多个逻辑相关的数据库的集合 |
| **分布式数据库管理系统（DDBMS）** | 管理分布式数据库的软件系统           |

* * *

二、分布式数据库的两大特性
--------------------

| 特性        | 含义                              |
| --------- | ------------------------------- |
| **数据透明性** | 用户像使用集中数据库一样，不需考虑数据的存储场地和操作执行场地 |
| **场地自治性** | 局部数据库可以独立于全局用户使用，允许局部应用         |

### 1. 数据透明性的三个层次

| 透明性层次     | 含义                          |
| --------- | --------------------------- |
| **分片透明性** | 用户不需考虑数据如何分片，系统自动完成分片到分段的映射 |
| **复制透明性** | 用户不需关心数据副本存储在哪，系统自动维护副本一致性  |
| **分布透明性** | 用户不需关心数据的存储场地和操作执行场地        |

### 2. 场地自治性的三个方面

| 自治性类型     | 含义                   |
| --------- | -------------------- |
| **设计自治性** | 局部DBMS能独立决定局部库的设计    |
| **通信自治性** | 局部DBMS能独立决定是否与其他场地通信 |
| **执行自治性** | 局部DBMS能独立决定局部操作的执行方式 |

三、分布式数据库的体系结构
-------------

### 1. 四层模式结构（我国标准）

```

       ↕ 映射

       ↕ 映射
局部概念层（局部逻辑模式）
       ↕ 映射
```

### 2. 组件架构（客户端/服务器）

| 组件            | 功能                    |
| ------------- | --------------------- |
| **应用处理器（AP）** | 处理跨场地请求、查询翻译、结果格式化    |
| **数据处理器（DP）** | 局部查询处理、恢复管理、存储管理、并发控制 |

四、分布式数据库的设计——数据分片
---------------------

### 1. 基本概念

| 术语                    | 定义            |
| --------------------- | ------------- |
| **分片（Fragmentation）** | 对全局数据的逻辑划分    |
| **片段（Fragment）**      | 划分后得到的子关系     |
| **分配（Allocation）**    | 将片段存储到物理场地的过程 |
| **分割（Partition）**     | 每个片段只存储在一个场地  |
| **复制（Replication）**   | 一个片段存储在多个场地   |

### 2. 分片的三原则

| 原则       | 含义              | 公式                     |
| -------- | --------------- | ---------------------- |
| **完备性**  | 所有全局数据必须映射到某个片段 | R = R₁ ∪ R₂ ∪ ... ∪ Rₙ |
| **可重构性** | 片段可以重新构成全局数据    | R = ψ(R₁, R₂, ..., Rₙ) |
| **不相交性** | 片段之间无重叠数据       | Rᵢ ∩ Rⱼ = ∅ (i≠j)      |

### 3. 分片的种类

| 类型       | 说明       | 操作    | 示例      |
| -------- | -------- | ----- | ------- |
| **水平分片** | 按元组划分（行） | 选择 σ  | 按部门分片   |
| **垂直分片** | 按属性划分（列） | 投影 π  | 按敏感信息分片 |
| **混合分片** | 水平+垂直组合  | σ + π | 先垂直再水平  |

### 4. 水平分片

**定义**：将关系的元组集划分成若干不相交的子集

**表示**：Rᵢ = σ_{条件ᵢ}(R)

**示例**：

```markdown
EMP {ENO, ENAME, SALARY, DNO}
E₁ = σ_{DNO=201}(EMP)   // 201部门
E₂ = σ_{DNO=202}(EMP)   // 202部门
E₃ = σ_{DNO≠201 ∧ DNO≠202}(EMP)  // 其他部门
```

### 5. 垂直分片

**定义**：按属性集将关系分成若干子集（主键必须在每个片段中）

**表示**：Rᵢ = π_{属性集ᵢ}(R)

**示例**：

```markdown
EMP {ENO, ENAME, BIRTH, SALARY, DNO}
E₁ = π_{ENO, ENAME, BIRTH}(EMP)
E₂ = π_{ENO, SALARY, DNO}(EMP)
```

### 7. 混合分片

**定义**：先垂直分片，再对垂直分片结果进行水平分片

**示例**：

```markdown
EMP {ENO, ENAME, BIRTH, SALARY, DNO}
E₁ = π_{ENO, ENAME, BIRTH}(EMP)
E₂ = π_{ENO, SALARY, DNO}(EMP)
E₂₁ = σ_{DNO=201}(E₂)
E₂₂ = σ_{DNO=202}(E₂)
E₂₃ = σ_{DNO≠201 ∧ DNO≠202}(E₂)
```

| 优势     | 代价      |
| ------ | ------- |

| 对比项  | 同步复制       | 异步复制      |

# 第6-2讲：分布式数据库中的查询优化

一、分布式查询优化的目标
--------------------

> **优化目标 = 局部执行代价 + 网络传输代价 最小**

| 代价类型       | 包含内容            |
| ---------- | --------------- |
| **局部执行代价** | I/O次数、CPU处理时间   |
| **网络传输代价** | 传输启动延迟 + 数据传输时间 |

**核心区别**：集中式数据库只关心I/O和CPU，分布式数据库还要关心**网络**。
二、优化内容

--------------

优化器需要决定：

1. **执行运算的次序**（先连接还是先选择？）

2. **执行每种运算的方法**（不同算法代价不同）

3. **访问哪个副本**（选择最近的场地，减少传输）

4. **在哪个场地执行运算**（让数据不动，运算动）

三、查询重写准则
--------------------

10条等价变换规则（交换律、结合律、分配律等），核心是两条**通用准则**：

| 准则      | 含义                       |
| ------- | ------------------------ |
| **准则1** | 尽可能将一元运算（σ、π）移到查询树底部（先做） |
| **准则2** | 利用一元运算缩减每个关系的大小，减少传输和I/O |

**关系运算代价排序**（从低到高）：

| 运算              | 时间复杂度               |
| --------------- | ------------------- |
| σ、π（不消重复）       | O(n)                |
| π（消重复）、GROUP BY | O(n log n)          |
| 连接（∞）、并、交、差     | O(n log n)          |
| **笛卡尔积（×）**     | **O(n²)** ← 最贵，尽量避免 |

* * *

五、查询处理四个层次（PDF第14-35页）
----------------------

这是分布式查询优化的完整流程：

```
用户查询（SQL）
1. 查询分解/查询映射
   将SQL转换为代数查询（关系代数树）
      ↓
2. 数据本地化
   将全局关系替换为片段（分片），生成片段查询树
      ↓
3. 全局查询优化
   确定最佳执行策略（在哪做、顺序、传输）
      ↓
4. 局部查询优化
   每个场地优化自己的子查询

```

##### 特点 ：非关系型的、分布式的、开源的、水平可扩展、数据模型灵活、高性能、高可用

### 3. MongoDB 的特点

* **面向集合存储**

* 使用 **BSON**（Binary JSON）格式存储

* 支持**复制**和**分片**

第7讲：XML数据传输格式

一、XML概述
-------

### 1. 什么是XML

> **XML（eXtensible Markup Language）**：可扩展标记语言

| 特性   | 说明                 |
| ---- | ------------------ |
| 全称   | 可扩展标记语言            |
| 设计目的 | **传输和存储数据**，而非显示数据 |
| 标签   | 没有被预定义，需要自行定义      |
| 特点   | 具有自我描述性            |
| 标准   | W3C推荐标准            |

### 2. XML vs HTML

| 对比项  | XML         | HTML     |
| ---- | ----------- | -------- |
| 设计目的 | **传输和存储数据** | 格式化并显示数据 |
| 标签   | 自定义         | 预定义      |
| 作用   | 数据交换        | 页面展示     |

### 3. XML的作用

| 作用            | 说明           |
| ------------- | ------------ |
| 把数据从HTML分离    | 数据独立存储，专注于显示 |
| 简化数据共享        | 不同系统间共享数据    |
| 简化数据传输        | 在不兼容系统间交换数据  |
| 配置文件          | 如config.xml  |
| 存储数据          | 充当小型数据库      |
| 创建新Internet语言 | 如RSS、SVG等    |

### 4. XML的特点

| 特点           | 说明                  |
| ------------ | ------------------- |

| **纯文本**      | 有能力处理纯文本的软件都可以处理XML |
| **可发明自己的标签** | 没有预定义标签             |
| **独立于软硬件**   | 数据传输工具              |

二、XML文档组成
---------

### 1. XML文档结构

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!-- 这是一个注释 -->
<note>
    <to>George</to>
    <from>John</from>
    <heading>Reminder</heading>
    <body>Don't forget the meeting!</body>
</note>
```

| 组成部分      | 说明                                               |
| --------- | ------------------------------------------------ |
| **XML声明** | `<?xml version="1.0" encoding="UTF-8"?>`，可选但建议添加 |
| **注释**    | `<!-- 注释内容 -->`                                  |
| **根元素**   | 文档最顶端的元素，**必须有且只有一个**                            |
| **元素**    | 从开始标签到结束标签的部分                                    |
| **属性**    | 在起始标签中添加，提供额外信息                                  |

### 2. XML树结构

* XML文档形成**树结构**

* 从**根**开始，扩展到**枝叶**

* 元素之间有**父子**和**同胞**关系

* * *

三、XML基本语法规则
-------------------

| 规则           | 说明        | 正确示例                        | 错误示例                 |
| ------------ | --------- | --------------------------- | -------------------- |
| **必须有根元素**   | 只有一个根元素   | `<root>...</root>`          | 多个根元素                |
| **标签必须关闭**   | 所有标签都要有关闭 | `<tag>...</tag>` 或 `<tag/>` | `<tag>`              |
| **大小写敏感**    | 区分大小写     | `<Message>`                 | `<message>`          |
| **必须正确嵌套**   | 标签不能交叉    | `<b><i>text</i></b>`        | `<b><i>text</b></i>` |
| **属性值必须加引号** | 单引号或双引号   | `<note date="2024">`        | `<note date=2024>`   |

### 实体引用（特殊字符转义）

| 实体引用     | 字符  | 含义  |
| -------- | --- | --- |
| `&lt;`   | `<` | 小于  |
| `&gt;`   | `>` | 大于  |
| `&amp;`  | `&` | 和号  |
| `&apos;` | `'` | 单引号 |
| `&quot;` | `"` | 双引号 |

> **注意**：在XML中，只有 `<` 和 `&` 确实是非法字符
> 
> **在 XML 文档中，只有 `<` 和 `&` 这两个字符，在**任何位置**出现（非转义形式）都会直接破坏 XML 语法结构，导致解析器报错。**
> 
> 而 `>`、`'`、`"` 只有在特定的“容易引起歧义”的上下文中才必须转义，在其他大多数地方可以直接写。

### CDATA（字符数据）

* 不会被解析器解析的文本

* 语法：`<![CDATA[ 不解析的内容 ]]>`

* 适用场景：特殊字符较多时

“不让它被解析”是为了**避免其中的 `<`、`&` 等字符被 XML 解析器误认为是标签或实体**，从而引发语法错误。

```xml
<knowledge>
    <![CDATA[1 < 2 && 2 < 3]]>
</knowledge>
```

四、良构与有效的XML
------------------

### 1. 良构（Well-Formed）XML

> 拥有**正确语法**的XML

**条件**：

* 有XML声明（可选）

* **有且只有一个根元素**

* 所有标签都关闭

* 大小写敏感

* 正确嵌套

* 属性值加引号

### 2. 有效（Valid）XML

> **良构** + 遵守**DTD**或**XML Schema**定义的规则

### 3. DTD（文档类型定义）

**作用**：定义合法的XML文档构建模块

**内部DOCTYPE声明**：

```xml
<!DOCTYPE 根元素 [
    <!ELEMENT 根元素 (子元素列表)>
    <!ELEMENT 元素名 (#PCDATA)>
    <!ATTLIST 元素名 属性名 属性类型 默认值>
]>

<!DOCTYPE students [
    <!-- 根元素 students，包含一个或多个 student 子元素 -->
    <!ELEMENT students (student+)>

    <!-- student 元素包含 name、age、gender 三个子元素，按顺序出现 -->
    <!ELEMENT student (name, age, gender)>

    <!-- name 元素内部是文本 -->
    <!ELEMENT name (#PCDATA)>
    <!-- age 元素内部是文本 -->
    <!ELEMENT age (#PCDATA)>
    <!-- gender 元素内部是文本 -->
    <!ELEMENT gender (#PCDATA)>

    <!-- student 元素有一个 id 属性，类型为 ID，必须填写 -->
    <!ATTLIST student id ID #REQUIRED>
]>
```

**外部DOCTYPE声明**：

```xml
<!DOCTYPE 根元素 SYSTEM "文件名.dtd">
```

**数量词**：

| 符号  | 含义          |
| --- | ----------- |
| `+` | 1次或多次（至少1次） |
| `?` | 0次或1次       |
| `*` | 任意次（0次或多次）  |

**PCDATA**：被解析的字符数据（parsed character data）

### 4. XML Schema（XSD）

* DTD的替代者

* **本身也是XML文档**

* 支持更丰富的数据类型

* 允许自定义数据类型

**XSD优势**：

| 优势      | 说明           |
| ------- | ------------ |
| 丰富的数据类型 | 支持整数、日期、布尔等  |
| 可读性强    | 本身就是XML      |
| 可扩展     | 针对未来需求可扩展    |
| 自定义数据类型 | 允许开发者定义自己的类型 |

五、XPath语言
---------

### 1. 什么是XPath

* **XML路径语言**

* 专门用来**查找XML数据内容**的语言

* 将文档视为**节点树**

### 2. XPath路径表达式

| 表达式  | 含义      |
| ---- | ------- |
| `/`  | 从根节点选取  |
| `//` | 从任意位置选取 |
| `.`  | 选取当前节点  |
| `..` | 选取父节点   |
| `@`  | 选取属性    |

```xpath
/bookstore/book      # bookstore下的所有book
//title              # 所有title元素
//book[@category]    # 所有带有category属性的book
//book[@category="WEB"]  # category属性为WEB的book
```
